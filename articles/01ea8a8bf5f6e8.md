## はじめに

2026年7月14日の [.NETラボ特別回](https://dotnetlab.connpass.com/event/398633/) では、英語セッション Why DevOps Still Matters in 2026 が予定されています。英語のままでも追える方は多いと思いますが、会場全員が内容に集中しやすいように、**英語音声から日本語字幕をリアルタイム生成するアプリ**を事前に用意することにしました🎤

今回の会場では、スライド用のプロジェクターとは別に、大きめのディスプレイへ字幕だけを表示する想定です。そのため必要だったのは、汎用的な会話アプリではなく、離れた位置からでも読みやすい全画面字幕アプリでした🖥️

実装の中核には Azure AI Speech / Speech Service のリアルタイム音声翻訳を使い、デスクトップ UI は WPF で構成しました。この記事では、イベント向けにどのような制約を置き、なぜこの技術選定がしっくり来たのかを、実装コードを交えながら紹介します。ベースにしたコードは [MainWindow.xaml.cs](https://github.com/tomokusaba/Trans/blob/main/MainWindow.xaml.cs) です📝

ここからは、「何を作ったか」だけでなく、「なぜその形が会場に合っていたのか」を順番に整理していきます🌟

## 今回のゴール / 本記事の目的

本記事で伝えたいポイントは次の 4 つです。

- 英語セッション向けに、英語音声 → 日本語テキストのリアルタイム字幕アプリをどう設計したか
- なぜ Azure AI Speech のリアルタイム翻訳を採用したのか
- なぜ表示基盤として WPF を選んだのか
- `TranslationRecognizer` を使った実装で、認識中表示と確定履歴をどう分けたのか

技術そのものの紹介に寄りすぎず、会場運用まで含めた「舞台裏」として読める形を目指します。まずは今回のイベント制約に対して何が必要だったのかを見ていきます🔍

## 背景 / なぜこのアプリが必要だったのか

今回の特別回では、英語セッションを日本語話者が会場で追いやすくすることが重要でした。ただし、スライド表示はすでにプロジェクターが担当するため、字幕は別ディスプレイに大きく、シンプルに、常時表示できる必要がありました💡

つまり要件は、単なる「翻訳できること」ではありません。低遅延であること、操作が簡単であること、遠くからでも読みやすいこと、そしてソースコードに固定値を埋め込まずに済むことが同時に必要でした⚙️

Azure AI Speech の公式ドキュメントでは、Speech Service はリアルタイムの多言語 speech-to-speech / speech-to-text translation をサポートすると説明されています。今回はその中から、イベント用途に最も素直な speech-to-text translation を選びました🌐

この切り分けを明確にしたことで、必要以上に機能を盛り込まず、字幕表示に集中した実装にできました。次は技術選定の理由です🎯

## 技術選定 / なぜ Azure AI Speech と WPF なのか

### Azure AI Speech を選んだ理由

今回欲しかったのは、汎用的な音声会話エージェントではなく、音声認識と翻訳をリアルタイムに返してくれる専用機能でした。Speech Service にはこの用途に合った SDK とイベントモデルがあり、`Recognizing` で中間結果、`Recognized` で確定結果を扱えるため、字幕 UI と相性がよかったです✨

また、今回のコードでは API キーではなく Microsoft Entra ID を使い、`DefaultAzureCredential` と `SpeechTranslationConfig.FromAuthorizationToken(...)` を組み合わせています。サブスクリプション キーをコードや設定ファイルへ直書きしない構成にできるのは大きな利点でした🔐

### GPT Realtime API 方式と比べてどうだったか

もちろん、汎用的な GPT Realtime API を使ってリアルタイム翻訳を組み立てる方法もあります。OpenAI の公式ドキュメントでも、Realtime API には翻訳専用セッションと音声エージェント向けセッションがあると整理されています。ただ、今回はそこまで広い設計自由度は不要で、まずは会場字幕として素直に動く構成を優先しました📉

| 観点 | Azure AI Speech リアルタイム翻訳 | GPT Realtime API 方式 |
|------|------|------|
| 🎯 主目的 | 音声翻訳そのものに素直 | 翻訳セッションや音声エージェントなど、扱える形の幅が広い |
| ⚡ 今回の設計方針との距離 | `TranslationRecognizer` 中心で組みやすい | 設計の自由度はあるが、今回はそこまで広げない判断にした |
| 🧭 今回の字幕用途との相性 | 字幕アプリに必要な責務へ集中しやすい | 今回は活かし切らない選択肢が多い |
| 🧩 今回の判断軸 | リアルタイム字幕に必要な機能へ集中しやすい | 柔軟性は高いが、今回は活かし切らない可能性が高い |
| 🏟️ 今回の適合度 | 採用 | 今回は不採用 |

今回は「会場で大きく安定表示する字幕」が主役だったので、Speech Service の専用性がそのままメリットになりました。では、なぜ UI 側は WPF だったのかを続けて説明します🪟

### WPF を選んだ理由

字幕を映す先は、スライド用プロジェクターではなく別の大画面ディスプレイです。この条件だと、ブラウザで開く汎用 UI よりも、全画面・ボーダーレス・大きな文字サイズを素直に作れるデスクトップ UI が向いていました🖥️

実際の UI では、ウィンドウを `WindowState="Maximized"`、`WindowStyle="None"` にし、認識中の英語と日本語を大きなフォントで左右表示しています。遠目でも読めることを優先できるので、WPF は今回かなり相性がよかったです👀

技術選定が定まったところで、実際に会場でどう見せる想定なのかを先に共有します🗺️

## できあがり像 / 動作イメージ

今回のアプリは、チャット画面のように多機能な UI を目指したわけではありません。英語の途中結果、日本語の途中翻訳、確定済みの履歴を遠くからでも追えることを優先しています👀

実装リポジトリを見ると、プロジェクトは [Trans.csproj](https://github.com/tomokusaba/Trans/blob/main/Trans.csproj) で `net9.0-windows` と `UseWPF=true` を採用しており、UI は [MainWindow.xaml](https://github.com/tomokusaba/Trans/blob/main/MainWindow.xaml) で `WindowState="Maximized"` と `WindowStyle="None"` を設定しています。つまり、最初から別ディスプレイへの全画面表示を前提にした構成です🖥️

| 表示領域 | 役割 | 実装上のポイント |
|------|------|------|
| 🇺🇸 認識中の英語 | 話者が今しゃべっている途中の文を表示 | `Recognizing` で更新 |
| 🇯🇵 翻訳中の日本語 | 日本語の途中翻訳を大きく表示 | `Recognizing` で更新 |
| 📝 履歴 | 確定した原文と訳文を保持 | `Recognized` で追加 |
| ⚙️ 設定 | リージョンとリソース ID を切り替え | `Expander` で折りたたみ |

画面の着地点が見えたので、次はアプリ全体の流れを図で見ます🧭

## アプリの全体構成

字幕アプリの処理フローは次のようになります。

```mermaid
flowchart LR
    A[会場マイク] --> B[AudioConfig.FromDefaultMicrophoneInput()]
    B --> C[TranslationRecognizer]
    D[DefaultAzureCredential] --> E[aad#resourceId#accessToken]
    E --> F[SpeechTranslationConfig]
    F --> C
    C -->|Recognizing| G[認識中の英語 / 日本語を大きく表示]
    C -->|Recognized| H[確定した英語 / 日本語を履歴に追加]
    G --> I[別ディスプレイへ全画面表示]
    H --> I
    J[local.settings.json / 環境変数] --> F
```

ポイントは、認識中の表示と確定済みの履歴を分けていることです。会場では「今しゃべっている途中の文」と「確定した字幕」を同じ扱いにすると読みにくくなるため、イベントごとに表示責務を分けました🔄

この構成を試す前に、必要な環境条件を先に確認します🧪

## 前提条件

- ✅ Windows 上で動く WPF アプリを前提にしています
- ✅ 実装リポジトリは [Trans.csproj](https://github.com/tomokusaba/Trans/blob/main/Trans.csproj) のとおり .NET 9 / WPF 構成です
- ✅ Azure AI Speech リソースを利用できること
- ✅ Microsoft Entra ID で Speech SDK を使う場合は、Speech リソースにカスタム サブドメインを設定済みであること
- ✅ Speech リソースに対して、今回は Cognitive Services User を IAM で割り当てています
- ✅ マイク入力と、字幕を映す別ディスプレイを用意できること

公式ドキュメントとしては、まず [音声翻訳とは何か](https://learn.microsoft.com/ja-jp/azure/ai-services/speech-service/speech-translation?WT.mc_id=DT-MVP-5004827)、Entra 認証まわりは [Microsoft Entra authentication with the Speech SDK](https://learn.microsoft.com/en-us/azure/ai-services/speech-service/how-to-configure-azure-ad-auth?pivots=programming-language-csharp&WT.mc_id=DT-MVP-5004827)、ロールは [Role-based access control for Speech resources](https://learn.microsoft.com/en-us/azure/ai-services/speech-service/role-based-access-control?WT.mc_id=DT-MVP-5004827) を見ると全体像をつかみやすいです📚

:::message
ここでいう **カスタム サブドメイン** は、「その Speech リソース専用の `https://<name>.cognitiveservices.azure.com/` というホスト名を持たせること」です。Microsoft Entra ID 認証では、この専用エンドポイントを前提にトークンを使ってアクセスするため、単に Speech リソースを作っただけでは足りず、専用 URL を有効にしておく必要があると理解すると分かりやすいです。
:::

この全体像を踏まえて、ここから実装の要点を見ていきます🛠️

## Step 1: 認証と翻訳設定を組む

今回の実装で一番「イベント向け」だと思っているのは、サブスクリプション キーをコードへ固定せずに Entra 認証で組んだ点です。`DefaultAzureCredential` でトークンを取得し、今回は `aad#{resourceId}#{accessToken}` 形式へ組み立ててから `SpeechTranslationConfig.FromAuthorizationToken(...)` を使っています🔑

```csharp
using Azure.Core;
using Azure.Identity;
using Microsoft.CognitiveServices.Speech.Translation;

var credential = new DefaultAzureCredential();
var tokenRequest = new TokenRequestContext(
    ["https://cognitiveservices.azure.com/.default"]);
var accessToken = await credential.GetTokenAsync(tokenRequest);

var authorizationToken = $"aad#{resourceId}#{accessToken.Token}";

var config = SpeechTranslationConfig.FromAuthorizationToken(
    authorizationToken, region);
config.SpeechRecognitionLanguage = "en-US";
config.AddTargetLanguage("ja");
```

`SpeechRecognitionLanguage = "en-US"` と `AddTargetLanguage("ja")` の組み合わせで、今回の要件である英語音声 → 日本語テキストを明示しています。複数ターゲット言語も可能ですが、会場字幕としてはまず 1 言語に絞る方が読みやすいと判断しました🇺🇸➡️🇯🇵

:::message
Microsoft Learn の [SpeechConfig.AuthorizationToken プロパティ](https://learn.microsoft.com/dotnet/api/microsoft.cognitiveservices.speech.speechconfig.authorizationtoken?view=azure-dotnet&WT.mc_id=DT-MVP-5004827) では、認可トークンには有効期限があり、期限前に更新が必要だと説明されています。既に作成済みの recognizer を使い続ける場合は、config だけでなく recognizer 側にも更新を反映する必要があります。今回のような短時間イベントならシンプルな構成でも回しやすいですが、**長時間の連続運用**を前提にするならトークン更新設計も追加したいところです。
:::

翻訳の軸が決まったので、次はマイク入力をつないでリアルタイム認識を開始します🎙️

## Step 2: マイク入力から `TranslationRecognizer` を起動する

音声入力は `AudioConfig.FromDefaultMicrophoneInput()` を使い、既定のマイクからそのまま取っています。今回の運用では専用の音声ファイル再生や複雑な入力切り替えは不要だったので、このシンプルさがありがたかったです🎧

```csharp
using Microsoft.CognitiveServices.Speech.Audio;
using Microsoft.CognitiveServices.Speech.Translation;

var audioConfig = AudioConfig.FromDefaultMicrophoneInput();
_recognizer = new TranslationRecognizer(config, audioConfig);

_recognizer.Recognizing += OnRecognizing;
_recognizer.Recognized += OnRecognized;
_recognizer.Canceled += OnCanceled;
_recognizer.SessionStarted += OnSessionStarted;
_recognizer.SessionStopped += OnSessionStopped;

await _recognizer.StartContinuousRecognitionAsync();
```

ここで重要なのは、イベントドリブンで UI を更新する設計にしていることです。`StartContinuousRecognitionAsync()` の後は、各イベントに応じて「今見せるべき字幕」を切り替えていきます⚡

つまり、字幕品質の体感は SDK 呼び出し 1 回ではなく、イベントの扱い方で大きく変わります。次がその本題です🧠

## Step 3: `Recognizing` と `Recognized` を分けて表示する

リアルタイム字幕では、途中結果と確定結果を混ぜると読みづらくなります。そこで `Recognizing` では現在進行中のテキストだけを大きく表示し、`Recognized` イベントのうち **`ResultReason.TranslatedSpeech` のときだけ** 確定した文を履歴へ追加する構成にしました📺

```csharp
private void OnRecognizing(object? sender, TranslationRecognitionEventArgs e)
{
    if (e.Result.Reason != ResultReason.TranslatingSpeech) return;

    e.Result.Translations.TryGetValue("ja", out var japanese);

    Dispatcher.Invoke(() =>
    {
        CurrentRecognitionText.Text = e.Result.Text;
        CurrentTranslationText.Text = japanese ?? "";
    });
}

private void OnRecognized(object? sender, TranslationRecognitionEventArgs e)
{
    if (e.Result.Reason != ResultReason.TranslatedSpeech) return;

    e.Result.Translations.TryGetValue("ja", out var japanese);

    Dispatcher.Invoke(() =>
    {
        if (!string.IsNullOrWhiteSpace(e.Result.Text))
        {
            RecognizedTextBox.AppendText(e.Result.Text + Environment.NewLine);
            TranslatedTextBox.AppendText((japanese ?? "") + Environment.NewLine);
        }

        CurrentRecognitionText.Text = "";
        CurrentTranslationText.Text = "";
    });
}
```

公式ドキュメントでも、音声検出中に中間トランスクリプションと翻訳結果が返り、最終結果も得られると説明されています。今回のアプリは、そのモデルを UI に反映した形です🧩

会場では話者の一文が長くなることもあるので、途中表示と確定履歴を分けたことが、読みやすさにかなり効きそうです。残る論点は、設定値をどう安全に扱うかです🗂️

## Step 4: 設定を外出しして運用しやすくする

今回の実装では、リージョンと Speech リソース ID を `local.settings.json` または環境変数から読み込めるようにしています。設定を切り替えやすく、ソースコードに固定値を埋め込まなくて済むためです🧰

```json
{
  "Speech": {
    "Region": "westus2",
    "ResourceId": "<your-speech-resource-id>"
  }
}
```

実装側では、[MainWindow.xaml.cs](https://github.com/tomokusaba/Trans/blob/main/MainWindow.xaml.cs) のとおり `local.settings.json` を優先し、なければ環境変数を見るようにしています。さらに UI 側には [MainWindow.xaml](https://github.com/tomokusaba/Trans/blob/main/MainWindow.xaml) の `Expander` を用意しているので、当日その場で確認や修正もしやすい構成です🔧

このあたりは「一時的なイベントアプリだから雑に作る」のではなく、現地で困らない最小限の運用性を確保するための工夫でした。続けて、当日の想定運用を短く確認します🎬

## 当日の想定運用 / 動作イメージ

当日は、登壇者の英語音声をマイクで拾い、アプリを表示した別ディスプレイへ字幕を全画面表示する想定です。画面には途中表示中の英語 / 日本語と、確定済みの履歴が同時に出るため、話者の現在地と少し前の文脈を両方追いやすくなります🎥

特に会場運用では、翻訳精度だけでなく「今しゃべっている文」と「確定した文」が視覚的に分かれていることが大切です。今回の UI はその点にかなり寄せているので、スライド用プロジェクターとは別の字幕専用画面として扱いやすいと考えています🪄

当日の使い方まで見えたところで、最後に実際に詰まりやすいポイントをまとめます⚠️

## ハマりどころ / 注意点

:::message alert
- Microsoft Entra 認証で Speech SDK を使う場合、Speech リソース側のカスタム サブドメイン設定が必要です  
- 今回は Speech リソースに Cognitive Services User を割り当てています  
- 必要な RBAC ロールがない場合は、Speech 利用時に認証 / 認可エラーになります  
- `Recognizing` の内容は揺れるので、履歴へそのまま積まず、`Recognized` で確定した文だけを残す方が読みやすいです  
- 会場用途では SDK の細かな最適化以上に、全画面表示・文字サイズ・色分けのほうが体感品質へ効きます
:::

技術的には小さなアプリでも、イベント運用まで含めると設計の優先順位がかなりはっきりしてきます。最後に今回の判断をまとめます📌

## まとめ

.NETラボ特別回に向けて作った今回のアプリは、英語セッションを日本語字幕で追いやすくするための専用ツールです。Azure AI Speech を使ったのは、リアルタイム翻訳という目的に対して実装が素直だったからで、WPF を使ったのは、別ディスプレイへ全画面で見やすく出すという会場制約に強かったからです🎯

また、`DefaultAzureCredential` と Entra 認証を使ったことで、サブスクリプション キーをコードや設定ファイルへ直書きせずに済みました。`Recognizing` と `Recognized` を分けた字幕表示も、単なる SDK 利用ではなく、会場体験に合わせた実装だったと感じています🔐

もし今後発展させるなら、固有名詞の表記ゆれ対策や、字幕レイアウトの微調整、ディスプレイ構成に応じた表示最適化を足していけそうです。まずは今回の .NETラボで、英語セッションをより追いやすくする一助になればうれしいです🙌
