# はじめに

Webアクセシビリティを説明するとき、どうしても「障害のある人のための特別な配慮」という文脈だけで語られがちです。しかし、WCAG 2.2 の原則 2 である **操作可能（Operable）** を丁寧に読むと、それはもっと広い話だと分かります🧭

操作可能とは、ユーザーインターフェースとナビゲーションが、**その人がその瞬間に取り得る操作方法で実行できること**を意味します。これは常設の障害だけではなく、骨折、捻挫、腱鞘炎、寝不足、発熱、薬の副作用、混雑した電車の中で片手しか使えない状況、そして光感受性てんかんのような持病まで含めて考える必要があるということです🩹

そして強調したいのは、**けがや病気は、誰にでも、いつ、どのタイミングで降りかかるか分からない**ということです。昨日まで何の不自由もなく操作できていた人が、明日には骨折や発熱、発作、強い片頭痛によって同じ操作を続けられなくなることは十分にあり得ます。アクセシビリティは「特定の誰かの事情」への配慮ではなく、将来の自分自身も含めた全員の不確実性に備える設計です🌧️

だからこそ、私たちが開発する Web システムはアクセシビリティに配慮する必要があります。それは善意のオプションではなく、**誰でも利用者になり得て、同時に誰でもある日突然使いにくさの当事者になり得る**以上、最初から品質要件として組み込むべきものです🛠️

Microsoft Learn の [アクセシビリティ対応 Web サイトの設計](https://learn.microsoft.com/ja-jp/microsoft-edge/accessibility/design) でも、キーボード操作、視覚的な判別性、多様な利用シナリオを前提に設計することが強調されています。W3C もまた、WCAG 2.2 の [Understanding the Four Principles of Accessibility](https://www.w3.org/WAI/WCAG22/Understanding/intro#understanding-the-four-principles-of-accessibility) において、アクセシブルなコンテンツは「知覚できる」「操作できる」「理解できる」「堅牢である」の四本柱で整理されると説明しています📚

本記事では、WCAG 2.2 の「操作可能」に含まれる主要な観点を、日常生活で遭遇しうるアクシデントや身体状況と結び付けながら整理します。特に、**一時的なけが**、**持病への配慮**、そして 1997 年のアニメ演出による大規模な失神・体調不良事件、いわゆる「ポケモンショック」が示した教訓までを含めて、なぜこの原則が技術的に重要なのかを考えます📝

# 操作可能とは「その人が今できる動作で使えること」

WCAG 2.2 における原則 2 の定義は、単に「ボタンが押せる」ではありません。キーボードで操作できること、時間制限が過剰でないこと、発作や身体的反応を誘発しないこと、現在位置が分かること、そしてタッチやポインタでも無理なく扱えることまで含んでいます⌨️

この原則が優れているのは、ユーザーを固定的な属性で分類しない点です。普段はマウスを使っている人でも、右手を骨折すればキーボード中心になるかもしれません。ふだんは素早く読める人でも、発熱時や強い疲労時には通知や制限時間に追い付けないかもしれません。アニメーションを好む人でも、片頭痛やてんかんの既往がある日は点滅や大きなモーションで体調を崩すかもしれません🌡️

ここで重要なのは、こうした変化が一部の人にだけ起きる例外ではないということです。転倒、感染症、睡眠不足、加齢、通院、服薬、育児や介護による疲労は、誰の生活にも入り込む可能性があります。だからこそ操作可能は、「今は健康だから大丈夫」という前提を崩すための原則でもあります🪶

つまり、操作可能は「障害者対応」のためだけでなく、**人間の状態が常に揺らぐことを前提にした設計原則**だと言えます⚙️

# 1. 片手しか使えない日は、2.1 キーボード操作可能が現実になる

WCAG 2.2 の [達成基準 2.1.1 キーボード](https://www.w3.org/WAI/WCAG22/Understanding/keyboard.html) は、すべての機能がキーボードインターフェースから操作可能であることを求めています。ただし、自由描画のように本質的に軌跡に依存する操作は例外です⌨️

この基準は、スクリーンリーダー利用者だけの話ではありません。たとえば次のような状況では、キーボードや限定的な入力方法が急に主役になります。

- 利き手を骨折してマウス操作が難しいとき🩼
- 指先を切ってドラッグ操作や長押しがつらいとき✋
- 赤ちゃんを抱えながら片手でスマートフォンを使うとき👶
- 通勤電車でつり革を持ちながら親指だけで操作するとき🚃

このとき、ドラッグでしか並べ替えできない UI、ホバーしないと現れないメニュー、Tab で移動できないカスタムコンポーネントは一気に使えなくなります。W3C の Understanding 2.1.1 でも、ポインタでできる多くの操作はキーボードでも実現できるべきだと説明されています🔍

特に問題になりやすいのは、見た目だけボタンで実体が div のような実装です。ネイティブ要素を使えばブラウザが持つキーボード操作、フォーカス管理、アクセシブルネームの多くを自動的に引き受けてくれますが、独自実装はその恩恵を捨てることになります⚠️

```html
<button type="button" class="menu-button" aria-expanded="false" aria-controls="global-nav">
	メニュー
</button>

<nav id="global-nav" hidden>
	<a href="/products">製品一覧</a>
	<a href="/pricing">料金</a>
	<a href="/support">サポート</a>
</nav>
```

```css
.menu-button:focus-visible,
#global-nav a:focus-visible {
	outline: 3px solid #005fcc;
	outline-offset: 3px;
}
```

この程度の実装でも、ネイティブ button と focus-visible を使うだけで、キーボード操作可能性は大きく改善します。複雑な UI を作る前に、まず「Tab だけで最後まで辿れるか」「Enter と Space で意図した操作ができるか」を確認するべきです🧪

# 2. 余裕がない日は、2.2 十分な時間が効いてくる

WCAG 2.2 の [ガイドライン 2.2 十分な時間](https://www.w3.org/WAI/WCAG22/Understanding/enough-time.html) は、ユーザーが読む、考える、入力する、確認するといった行為に十分な時間を与えることを求めています⏳

この要件も、常設の障害だけを対象にしていません。W3C は、身体的に反応に時間がかかる人、読むのに時間がかかる人、低視力の人、支援技術を利用していて操作に時間がかかる人などを例示しています。これを日常生活に引き寄せると、次のような状況がすぐ思い浮かびます🧠

- 高熱や強い疲労で認知処理が遅いとき🤒
- 片手入力でフォーム入力に通常の倍以上かかるとき📱
- 服薬の影響で集中力や反応速度が落ちているとき💊
- 途中で子どもに呼ばれたり、駅で乗り換えをしたりして中断が入るとき⏸️

ログイン画面や決済画面で、何の警告もなくセッションが切れる設計はこの文脈で非常に厳しいものになります。フォームを半分入力した時点で突然タイムアウトし、入力内容も失われるなら、それは単に「不便」ではなく、操作可能性を破壊しています💥

時間制限が本当に必要なら、少なくとも延長、警告、再開可能性を提供するべきです。次のような実装は、その最小構成として有効です🔧

```javascript
const SESSION_TIMEOUT_MS = 10 * 60 * 1000;
const WARNING_BEFORE_MS = 60 * 1000;

let timeoutId;
let warningId;

function startSessionTimer() {
	clearTimeout(timeoutId);
	clearTimeout(warningId);

	warningId = setTimeout(() => {
		const extend = window.confirm(
			"まもなくセッションが期限切れになります。作業を続ける場合はOKを押してください。"
		);

		if (extend) {
			startSessionTimer();
			return;
		}

		saveDraft();
	}, SESSION_TIMEOUT_MS - WARNING_BEFORE_MS);

	timeoutId = setTimeout(() => {
		saveDraft();
		location.href = "/session-expired";
	}, SESSION_TIMEOUT_MS);
}
```

ここで重要なのは、期限の存在そのものより、**ユーザーが主導権を持てること**です。延長できること、下書きが保存されること、戻って再開できることが、操作可能を支えます💾

# 3. 持病や体質への配慮は、2.3 発作と身体的反応で明文化されている

「操作可能」の中で見落とされやすいのが、[ガイドライン 2.3 発作と身体的反応](https://www.w3.org/WAI/WCAG22/Understanding/seizures-and-physical-reactions.html) です。このガイドラインは、**発作や身体的な反応を引き起こすことが知られている方法でコンテンツを設計してはならない**と明言しています⚡

これは、光感受性てんかんのある人だけの問題ではありません。W3C はこのガイドラインを「発作や身体的反応を引き起こすことが知られている方法でコンテンツを設計しないこと」と説明しており、強い点滅や過剰なモーションは、一部の人にとって安全に利用し続けられない要因になり得ます。つまり「見た目が派手で楽しい演出」が、別の人にとってはその場で利用を中断せざるを得ない危険な刺激になり得るということです🌪️

W3C の Understanding 2.3 では、1997 年に日本で放送されたアニメが、700 人を超える子どもたちの搬送や発作を引き起こした事例に触れています。これは一般に「ポケモンショック」と呼ばれる事件で、赤と青の強い点滅演出が問題になりました。さらに PubMed に掲載されている報告 [Photosensitive seizures provoked while viewing "pocket monsters," a made-for-television animation program in Japan](https://pubmed.ncbi.nlm.nih.gov/9860071/) でも、深い赤と明るい青の高速な点滅が原因として示され、規制や技術ガイドラインの必要性が結論付けられています📄

ここで重要なのは二つあります。

第一に、**警告を出せばよいわけではない**という点です。W3C は、この種の警告は特に子どもには見落とされやすく、十分に機能しないと述べています。加えてテレビのようなメディアでは、視聴者が必ずしもコンテンツを最初から見るとは限りません。番組の途中から視聴を始めた人は、冒頭の注意書きそのものを見る機会がない可能性があります。つまり、警告に安全性を委ねる設計自体が不十分であり、危険な刺激は、そもそも設計段階で作らないことが原則です🚫

第二に、**本人が自分の感受性を認識していない場合がある**という点です。W3C は、多くの人がこの種の発作特性を、実際に発作が起きるまで知らないと指摘しています。つまり、「必要な人は自分で避けるはずだ」という前提自体が破綻しています🧷

アニメーションや点滅演出を扱うプロダクトでは、最低でも次の観点が必要です。

- 1 秒間に 3 回を超える点滅を避けること💡
- 赤色を含む高コントラストな点滅を避けること🟥
- 自動再生のモーションや背景動画を止められること⏹️
- prefers-reduced-motion を尊重すること🎚️

```css
@media (prefers-reduced-motion: reduce) {
	*,
	*::before,
	*::after {
		animation-duration: 0.01ms !important;
		animation-iteration-count: 1 !important;
		transition-duration: 0.01ms !important;
		scroll-behavior: auto !important;
	}

	.parallax,
	.auto-carousel,
	.background-pulse {
		animation: none !important;
		transform: none !important;
	}
}
```

この CSS だけで点滅リスクを完全に防げるわけではありませんが、「激しい演出を好まない、あるいは耐えられない」ユーザーに逃げ道を提供できます。派手な演出を足す前に、その演出が**誰かの離脱要因**ではなく**身体的リスク**になっていないかを問うべきです🛑

# 4. 迷いやすい日は、2.4 ナビゲーション可能が効いてくる

操作可能は、単に入力できることだけでは成立しません。今どこにいるのか、どこへ移動できるのか、どの要素にフォーカスがあるのかが分からなければ、事実上操作できないからです🗺️

WCAG 2.2 の [ガイドライン 2.4 ナビゲーション可能](https://www.w3.org/WAI/WCAG22/Understanding/navigable.html) では、スキップリンク、ページタイトル、フォーカス順序、見出し、フォーカス可視化、フォーカス非隠蔽といった要件が整理されています。W3C も、このガイドラインの意図を、ユーザーが必要なコンテンツを見つけ、現在位置を把握し、移動先を理解できるようにすることだと説明しています🧠

たとえば、寝不足の朝や体調不良の日は、普段なら一瞬で理解できる画面でも認知負荷が高く感じられます。そのとき、フォーカスリングが見えない、モーダルを開いたら背景にフォーカスが逃げる、ヘッダーが固定表示されてフォーカスされた要素が隠れるといった実装は、操作可能性を著しく下げます😵

特に WCAG 2.2 で追加された 2.4.11 と 2.4.13 は重要です。フォーカスが見えるだけでなく、**隠れず、十分に視認できること**まで求められるようになりました。これは「Tab は動くが、今どこにいるか分からない」という半端な実装を許さない方向への前進です📍

```html
<a class="skip-link" href="#main">本文へスキップ</a>

<header>...</header>
<main id="main" tabindex="-1">
	<h1>注文内容の確認</h1>
	<!-- main content -->
</main>
```

```css
.skip-link {
	position: absolute;
	top: 8px;
	left: 8px;
	transform: translateY(-200%);
	background: #ffffff;
	color: #111111;
	padding: 12px 16px;
	z-index: 1000;
}

.skip-link:focus {
	transform: translateY(0);
}
```

スキップリンクは古典的ですが、繰り返しナビゲーションを毎回通らずに本文へ移動できるため、キーボード利用者にも、認知的に疲れているユーザーにも有効です。操作可能は、移動距離と迷子の時間を減らす設計でもあります🚪

# 5. 指先の精度が落ちる日は、2.5 入力モダリティが支える

最後に、スマートフォン中心の設計で非常に重要なのが [ガイドライン 2.5 入力モダリティ](https://www.w3.org/WAI/WCAG22/Understanding/input-modalities.html) です。ここでは、複雑なジェスチャーの代替、ポインタキャンセル、ドラッグ操作の代替、ターゲットサイズの確保などが求められます📱

これは、手指に震えがある人や運動機能障害のある人だけでなく、日常の「うまく押せない瞬間」にそのまま効きます。

- 手をぶつけて細かい操作がしづらいとき🤕
- 冬場に手がかじかんでいるとき🥶
- 片手の親指だけで画面端を操作しているとき👍
- 歩行中や乗り物の揺れでタップ精度が落ちるとき🚌

W3C の [達成基準 2.5.8 ターゲットサイズ（最低限）](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html) は、タッチターゲットを 24 x 24 CSS ピクセル以上、もしくは十分な間隔付きにすることを求めています。W3C 自身も、この基準はモバイル利用者、手の震えがある人、公共交通機関の揺れの中で操作する人、片手操作の人などに利益があると説明しています📐

つまり、小さすぎる閉じるボタン、隣接しすぎたアイコン、ドラッグ必須の UI は、特定の障害だけでなく、**日常の不安定な操作環境全般**に弱い設計だということです⚠️

```css
.icon-button {
	min-width: 24px;
	min-height: 24px;
	padding: 8px;
}

.toolbar {
	display: flex;
	gap: 8px;
}
```

見た目のアイコンが 16px であっても、押下領域を 24px 以上に確保し、隣接要素との間隔を取れば、誤タップの確率は大きく下がります。デザイン上のミニマルさと、操作領域の十分さは両立できます🎯

# 実装時に確認したい、操作可能のチェックポイント

ここまでの話を実装観点でまとめると、操作可能を損なう原因は意外と共通しています。入力方法を一つに限定し、時間に余裕を与えず、視覚的演出を優先しすぎ、現在位置の手掛かりを消し、小さなターゲットに操作を押し込むことです🧱

レビューやQAでは、少なくとも次の問いを持っておくと有効です。

1. マウスやタッチを使えない前提でも、主要機能は完了できるか🔎
2. 時間切れや自動更新があっても、ユーザーは延長・停止・再開できるか⏱️
3. 点滅、強いモーション、自動再生は、本当に必要か。停止手段はあるか⚡
4. Tab を押したとき、今どこにいるかが常に分かるか📍
5. スマートフォンの片手操作でも、誤タップせずに押せる大きさと間隔があるか📱

もしこれらの問いのどれかに曖昧さが残るなら、その UI はまだ「操作可能」とは言い切れません。アクセシビリティの問題は、しばしば例外的なケースとして後回しにされますが、実際には**不安定な人間に対して不安定な UI を出してしまっている**だけのことが多いのです🪞

# おわりに

WCAG の「操作可能」は、単なる入力デバイス対応のチェックリストではありません。それは、人がけがをすること、疲れること、加齢すること、持病とともに暮らすこと、そしてある日突然これまで通りに操作できなくなることを前提にした設計思想です🌍

片手しか使えない日にはキーボード操作性が生き残りを分けます。集中できない日には時間制限の設計が効きます。光感受性てんかんや片頭痛のある人にとっては、点滅や過剰なアニメーションそのものが危険になります。そして、ポケモンショックのような事件は、その危険が決して抽象論ではないことを示しました⚠️

アクセシビリティを高めることは、特殊な利用者を想像することではありません。**けがも病気も、誰にでも、いつ何時起こるか分からない**という、ごく現実的な事実を受け入れることです。人間はいつでも操作能力を失いうるし、その変化は予告なく訪れます。その前提に立てば、「操作可能」は親切さの付加機能ではなく、プロダクト品質そのものの条件だと見えてきます🧩

そして、その配慮は一つひとつが小さく見えても無意味ではありません。キーボードで操作できるようにすること、点滅を避けること、時間切れの前に警告すること、押しやすいサイズを確保すること、フォーカスを見えるようにすること。そうした実装上の積み重ねが、結果として**すべての人が使えるシステムを少しずつ増やしていく**ことにつながります🌱
