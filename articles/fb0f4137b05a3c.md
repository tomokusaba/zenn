# はじめに

Windows には、追加のソフトウェアを入れなくても使えるアクセシビリティ機能が数多く用意されています。

ただ、機能名を一覧で眺めるだけだと、それぞれがどの場面で役立つのかが少し見えにくくなります。

そこでこの記事では、Windows に標準搭載されているアクセシビリティ機能を、次の 3 つの観点で整理して紹介します。

1. 操作系に関する機能
2. 入力補助に関する機能
3. 表示に関する機能

なお、機能名や設定画面の表記は Windows 10 と Windows 11 で一部異なることがありますが、考え方そのものは共通しています。

# まず押さえたい前提

Microsoft Learn の [Accessibility information for IT professionals](https://learn.microsoft.com/windows/configuration/accessibility/) では、Windows のアクセシビリティ機能は、ユーザーが自分に合った形でコンピューター体験を調整するための仕組みとして紹介されています。

また、[Accessibility overview](https://learn.microsoft.com/windows/apps/design/accessibility/accessibility-overview#assistive-technology) では、スクリーン リーダー、画面拡大、高コントラスト、音声認識、スクリーン キーボードなどが、OS 側で提供される代表的な支援技術として整理されています。

加えて、Microsoft Support の [Windows アクセシビリティ機能を確認する](https://support.microsoft.com/windows/discover-windows-accessibility-features-8b1068e6-d3b8-4ba8-b027-133dd8911df9) では、Windows のアクセシビリティ機能が、ビジョン、聴覚、四肢と身体能力、フォーカスなどの観点で整理されています。

この 2 つを踏まえると、Windows のアクセシビリティ機能は単なる「特別な設定」ではなく、操作方法、入力方法、見え方を利用者ごとに最適化するための標準機能群だと捉えられます。

# 操作系に関する機能

ここでいう「操作系」は、PC の状態を把握しながら、アプリや OS を進めていくための機能です。

マウスや視覚中心の操作が難しい場合でも、読み上げ、音声、視線など別の手段で Windows を扱えるようにする機能がここに入ります。

## Narrator

Narrator は Windows に組み込まれているスクリーン リーダーです。

Microsoft Learn の [Accessibility overview](https://learn.microsoft.com/windows/apps/design/accessibility/accessibility-overview#assistive-technology) では、Narrator は Windows に含まれるスクリーン リーダーとして説明されており、タッチ モードにも対応するとされています。

Microsoft Support の [ナレーターの詳細なガイド](https://support.microsoft.com/windows/complete-guide-to-narrator-e4397a0d-ef4f-b386-d8ae-c172f109bdb1) でも、Narrator は Windows に組み込まれた画面読み上げアプリであり、アプリの利用や Web の閲覧に使えることが案内されています。

視覚情報を音声に変換してくれるため、画面を見づらい場合だけでなく、表示を見続ける負荷を減らしたい場面でも有効です。

また、[Accessibility testing](https://learn.microsoft.com/windows/apps/design/accessibility/accessibility-testing#accessibility-testing-procedures) では、キーボードを使って UI を移動しながら、要素の名前、状態、種類が読み上げられるかを確認する手順が紹介されています。

つまり Narrator は「読むための機能」であると同時に、「今どこにいて、何を操作しようとしているか」を把握するための操作支援でもあります。

## Voice access と音声操作

声で PC を操作したい場合には、音声系の機能が大きな助けになります。

Microsoft Learn の [What's new in Windows 11, version 22H2](https://learn.microsoft.com/windows/whats-new/whats-new-windows-11-version-22h2#windows-accessibility) では、Windows 11 22H2 のアクセシビリティ改善として、system-wide live captions、voice access、より自然な Narrator 音声が挙げられています。

さらに、[Speech, voice, and conversation in Windows 11 and Windows 10](https://learn.microsoft.com/windows/apps/develop/speech) では、音声認識、ディクテーション、音声合成といった仕組みが、マウスやキーボードの代替または補完となる自然な操作手段として整理されています。

このため、Windows 11 では voice access のような新しい音声操作が強化される一方、Windows 全体としては従来から音声認識やディクテーションの流れが積み重ねられてきた、と理解すると整理しやすいです。

手を使った操作が難しい場面では、音声でアプリの切り替えやテキスト入力を補えるため、「入力補助」だけでなく「操作そのものの代替手段」として非常に重要です。

## Eye control と視線入力

視線で操作する機能も、操作系の代表例です。

Microsoft Learn の [Gaze interactions and eye tracking in Windows apps](https://learn.microsoft.com/windows/uwp/ui-input/gaze-interactions) では、視線入力は ALS など神経筋系の疾患を持つ人に特に有効な支援技術であり、Windows 10 Fall Creators Update で Eye control が導入されたことが説明されています。

同ページでは、Eye control によって画面上のポインター操作、オンスクリーン キーボードの利用、テキスト読み上げによるコミュニケーションが可能になるとされています。

また、Microsoft Support の [Windows アクセシビリティ機能を確認する](https://support.microsoft.com/windows/discover-windows-accessibility-features-8b1068e6-d3b8-4ba8-b027-133dd8911df9) でも、視線制御は四肢と身体能力に関わる支援機能として案内されています。

操作の中心を「手」から「目線」へ移せる点で、Eye control は Windows のアクセシビリティを象徴する機能の 1 つだと思います。

# 入力補助に関する機能

次は、文字入力やショートカット入力、ポインター入力を補助する機能です。

「入力できること」はアプリ利用の前提なので、ここが整うだけで Windows の使いやすさは大きく変わります。

## オンスクリーン キーボード

物理キーボードを使いにくい場合、まず候補に挙がるのがオンスクリーン キーボードです。

Microsoft Learn の [On-screen keyboard](https://learn.microsoft.com/windows/iot/iot-enterprise/os-features/on-screen-keyboard#enable-on-screen-keyboard) では、Windows のオンスクリーン キーボードは物理キーボードの代わりに使える標準機能であり、マウスなどのポインティング デバイスでもキー入力できると説明されています。

さらに同ページでは、クリック入力だけでなく、ホバー入力、スキャン入力、テキスト予測も利用できることが示されています。

Microsoft Support の [スクリーン キーボード (OSK) を使用して入力する](https://support.microsoft.com/windows/use-the-on-screen-keyboard-osk-to-type-ecbb5e08-5b4e-d8c8-f794-81dbf896267a) も、OSK が物理キーボードの代替として使えることを案内しています。

単に「画面にキーボードを出す」だけではなく、入力方法そのものを利用者に合わせて変えられる点が重要です。

## StickyKeys、FilterKeys、ToggleKeys、MouseKeys

キーボードやポインターの扱いを細かく補助する機能群も、Windows には標準で用意されています。

Microsoft Learn の [Built-in Accessibility Features](https://learn.microsoft.com/windows/win32/winauto/built-in-accessibility-features) では、次のような代表機能が紹介されています。

1. StickyKeys: 複数キー同時押しではなく、順番に押してショートカットを成立させる機能
2. FilterKeys: 連打や短すぎるキー入力を抑制し、入力を安定させる機能
3. ToggleKeys: Caps Lock、Num Lock、Scroll Lock の切り替えを音で把握しやすくする機能
4. MouseKeys: テンキーでマウス ポインターを操作できる機能

同ページにはホットキーも整理されており、たとえば StickyKeys は Shift キー 5 回、FilterKeys は右 Shift キー長押し、MouseKeys は Left Alt + Left Shift + Num Lock で呼び出せます。

これらは派手な機能ではありませんが、ショートカット操作や細かいポインター操作の負担をかなり下げてくれます。

この中でも、日常的なキーボード操作に直結しやすいのが StickyKeys です。

### StickyKeys をもう少し詳しく見る

Microsoft Learn の [STICKYKEYS structure (winuser.h)](https://learn.microsoft.com/windows/win32/api/winuser/ns-winuser-stickykeys) では、StickyKeys は SHIFT、CTRL、ALT などの修飾キーを同時押しではなく順番に押しても成立させられる機能として説明されています。

さらに同ページでは、StickyKeys の動作として次の 2 段階が説明されています。

1. 修飾キーを 1 回押すと latch され、次の非修飾キー入力まで押下状態を保持する
2. 修飾キーを 2 回押すと lock され、3 回目に押すまで保持し続ける

この挙動を日本語で言い換えると、StickyKeys は「一時的に押したままにする」と「しばらく固定する」を切り替えながら使える機能です。

たとえば通常のキーボード操作では、コピーは Ctrl + C、ウィンドウ切り替えは Alt + Tab のように同時押しが必要です。

しかし StickyKeys を有効にすると、次のように順番押しで操作できます。

1. Ctrl を押す
2. C を押す

これで Ctrl + C と同じ意味になります。

Alt + Tab も同様で、Alt を押してから Tab を押せばウィンドウ切り替えとして扱えます。

さらに Alt を 2 回押して lock した状態にすると、Alt を押し続けなくても Tab を繰り返してウィンドウを選びやすくなります。

### StickyKeys が役立つ場面の例

StickyKeys が特に有効なのは、同時押しそのものが負担になる場面です。

たとえば次のようなケースでは効果がわかりやすいです。

1. 片手でしかキーボードを操作しにくいとき
2. 指先の細かい力加減が難しく、複数キーを同時に押すと別のキーまで触れてしまうとき
3. ノート PC の小さなキーボードで、Ctrl、Shift、Alt を使うショートカットが連続するとき
4. 一時的なけがや疲労で、長く押し続ける動作がつらいとき

たとえばファイルを複数選択するときは Shift を押しながら矢印キーを使う場面があります。

StickyKeys があれば、先に Shift を押して保持し、その後に矢印キーを押して範囲選択できます。

同じように、Ctrl を使った複数選択やショートカットも、押しっぱなしを前提にしなくて済みます。

### StickyKeys を紹介するときに押さえたい点

Microsoft Learn の [Built-in Accessibility Features](https://learn.microsoft.com/windows/win32/winauto/built-in-accessibility-features) では、StickyKeys は Shift キーを 5 回押すことで呼び出せるホットキーを持つことが示されています。

また、[STICKYKEYS structure (winuser.h)](https://learn.microsoft.com/windows/win32/api/winuser/ns-winuser-stickykeys) には、音によるフィードバック、ホットキーの有効化、修飾キーの lock 動作など、StickyKeys の状態管理に関するフラグが定義されています。

このことからも、StickyKeys は単純なショートカット簡略化ではなく、「修飾キーの状態をわかりやすく保ちながら操作を成立させる」ための仕組みとして設計されていることがわかります。

## 音声による入力補助

音声は操作だけでなく、入力補助の手段としても重要です。

先ほど触れた [Speech, voice, and conversation in Windows 11 and Windows 10](https://learn.microsoft.com/windows/apps/develop/speech) では、speech recognition や dictation が、従来のキーボード入力を補完または置き換える手段として説明されています。

長文入力が続く場面や、手指の疲労を避けたい場面では、音声入力を併用するだけでも作業の継続しやすさがかなり変わります。

# 表示に関する機能

最後は、画面の見え方や読みやすさを調整する機能です。

アクセシビリティというと読み上げや音声入力が注目されがちですが、実際には「見える」「追える」「識別できる」を支える表示系の調整も非常に重要です。

## Magnifier

Microsoft Learn の [Accessibility (Design basics)](https://learn.microsoft.com/windows/win32/uxguide/inter-accessibility#design-concepts) では、Magnifier は弱視のユーザーを支援するために、画面上の項目を元の 2 倍から 16 倍まで拡大できる機能として説明されています。

さらに同ページでは、マウス位置、キーボード フォーカス、テキスト編集位置を追従対象にできることにも触れられています。

そのため Magnifier は、単に拡大するだけではなく、「どこを見ればよいか」を追いやすくする機能でもあります。

## Contrast themes

高い視認性が必要な場合には、コントラスト テーマが有効です。

Microsoft Learn の [Contrast themes](https://learn.microsoft.com/windows/apps/design/accessibility/high-contrast-themes) では、コントラスト テーマは少ない色数で高いコントラスト比を確保し、UI 要素を見やすくして、目の疲れを減らし、文字の可読性を高めるためのものだと説明されています。

また、同ページでは、Settings > Accessibility > Contrast themes から有効化やカスタマイズができること、Left Alt + Shift + Print Screen で素早く切り替えられることも紹介されています。

Microsoft Support の [Windows で色のコントラストを変更する](https://support.microsoft.com/windows/change-color-contrast-in-windows-fedc744c-90ac-69df-aed5-c8a90125e696) でも、ユーザー向けの設定手順が案内されています。

背景と文字の差が取りづらい環境や、まぶしさを避けたい環境では、こうしたテーマ変更が作業継続性に直結します。

## 文字サイズ、カラーフィルター、カーソル関連

表示の調整は、拡大や高コントラストだけではありません。

Microsoft Learn の [Accessibility checklist](https://learn.microsoft.com/windows/apps/design/accessibility/accessibility-checklist) では、Windows にはテキスト サイズの調整や表示倍率の変更など、読みやすさを高めるための仕組みが用意されていることが説明されています。

さらに、[Windows Backup for Organizations settings catalog](https://learn.microsoft.com/windows/configuration/windows-backup/catalog#accessibility) の Accessibility セクションを見ると、Make text bigger、Magnifier、Color filters、Contrast themes、Mouse pointer and touch、Text cursor など、表示や追従に関する設定項目が広く用意されていることがわかります。

この中でも、色覚特性に直接関わる設定として重要なのがカラーフィルターです。

### カラーフィルターと色覚障害

Microsoft Learn の [Color in Windows](https://learn.microsoft.com/windows/apps/design/signature-experiences/color#color-principles) では、赤緑系の色覚障害があるユーザーは赤と緑の要素を区別しづらく、色の組み合わせを唯一の識別手段にしてはいけないと説明されています。

また、[Developing inclusive Windows apps](https://learn.microsoft.com/windows/apps/design/accessibility/developing-inclusive-windows-apps#color-and-contrast) でも、色覚障害のあるユーザーは特定の色を見分けにくいため、状態や意味を色だけで伝えないことが重要だと述べられています。

これは逆に言うと、Windows の側で色の見え方を調整できることにも意味があるということです。

カラーフィルターは、画面全体の色変換を通じて、区別しづらい色同士の差を利用者にとって把握しやすい方向へ寄せる機能です。

Microsoft Support の [Windows でカラー フィルターを使用する](https://support.microsoft.com/windows/use-color-filters-in-windows-43893e44-b8b3-2e27-1a29-b0c15ef0e5ce) でも、カラー フィルターは画面上のカラー パレットを変更し、色だけが異なる項目を区別するのに役立つと説明されています。

色覚障害と一口に言っても見え方は一様ではありません。

Microsoft Learn の [Color](https://learn.microsoft.com/windows/win32/uxguide/vis-color#design-concepts) では、代表例として Protanopia、Deuteranopia、Tritanopia のような見え方の違いが紹介されています。

さらに、[Emulate vision deficiencies](https://learn.microsoft.com/microsoft-edge/devtools/accessibility/emulate-vision-deficiencies) では、色覚特性の例として Protanopia、Deuteranopia、Tritanopia、Achromatopsia が挙げられており、同じ「色が見分けづらい」でも困り方が異なることがわかります。

そのため、カラーフィルターの価値は「すべての色覚障害を 1 つの設定で解決する」ことではなく、利用者ごとの見え方に近づけて、情報の差を取り戻しやすくする点にあります。

### どんな場面で役立つか

たとえば、次のような UI は色覚特性の影響を受けやすい典型例です。

1. エラーを赤、成功を緑だけで示している画面
2. グラフの系列を色だけで区別しているダッシュボード
3. オンとオフ、有効と無効を色だけで表している設定画面
4. 地図、ヒートマップ、進捗表示のように色差が意味そのものになっている画面

Microsoft Learn の [Color blindness](https://learn.microsoft.com/surface/inclusive-tech-lab/vision-color-blindness) では、色が唯一の情報伝達手段になっていると、その色を識別できない人は情報を解釈できなくなると説明されています。

たとえば赤と緑だけで状態を示している一覧画面では、色の差がつかないと「正常」と「異常」が見分けにくくなります。

このときカラーフィルターを有効にすると、色差の知覚が改善されて、少なくとも状態の差を追いやすくなることがあります。

### ただし、カラーフィルターだけに依存してはいけない

ここは記事として強調しておきたい点です。

Microsoft Learn の [Designing inclusive software for Windows](https://learn.microsoft.com/windows/apps/design/accessibility/designing-inclusive-software#practical-design-steps) では、色覚障害のあるユーザーに配慮するため、色だけで状態や意味を伝えないことが重要だとされています。

つまり、カラーフィルターは利用者側の助けにはなりますが、アプリや OS の情報設計そのものを置き換えるものではありません。

理想的には、次の 2 つが両方必要です。

1. Windows 側でカラーフィルターやコントラスト設定を使って、自分に合う見え方へ調整できること
2. アプリ側でも、アイコン、ラベル、記号、テキスト、形状差などを併用して、色だけに依存しないこと

実際の利用では、「カラーフィルターを有効にすると見やすくなる」場面は多い一方で、それでも区別が難しい配色は残ります。

だからこそ Windows がカラーフィルターを持っていることと、ソフトウェアが色以外でも意味を伝えることは、セットで考える必要があります。

### この節で押さえたいこと

実際の利用場面では、「画面全体を大きくしたい」のか、「文字だけ大きくしたい」のか、「ポインターを見失わないようにしたい」のか、「色の差を判別しやすくしたい」のかで必要な設定は変わります。

Windows が複数の調整手段を持っているのは、この違いに対応するためです。

# まとめ

Windows のアクセシビリティ機能は、単に支援技術を並べたものではなく、利用者ごとに PC の使い方そのものを再設計できる仕組みだと感じます。

今回の 3 分類でまとめると、役割は次のように整理できます。

1. 操作系: Narrator、Voice access、Eye control など、PC を進めるための手段を増やす機能
2. 入力補助: オンスクリーン キーボード、StickyKeys、FilterKeys、MouseKeys、音声入力など、入力方法を調整する機能
3. 表示系: Magnifier、Contrast themes、文字サイズやカラーフィルターなど、見え方を調整する機能

アクセシビリティの話は、ともすると「特定の誰かのための機能」として語られがちです。

ですが実際には、疲労、騒音、まぶしさ、一時的なけが、手が離せない状況など、誰にでも起こりうる条件変化に対して Windows を使い続けやすくする仕組みでもあります。

まずは自分の PC の Settings > Accessibility を一度開いてみるだけでも、Windows の設計思想がかなり見えてくるはずです。
