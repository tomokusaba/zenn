## はじめに

2026年2月2日、ジャストシステムはATOK for Windows（Tech Ver.36）のアップデートを公開し、**Arm版Windows 11へのネイティブ対応（ARM64）** を正式に発表しました。

https://www.atok.com/announce/2026/20260202_1.html

Copilot+ PCの普及が進む中、日本語入力システムという「OSの根幹に関わるソフトウェア」がARM64ネイティブで動作するようになったことは、Arm版Windows環境の成熟を示す大きな一歩です。

## Arm版Windowsとかな漢字変換の課題

### x86/x64エミュレーションの限界

Arm版Windows 11には、x86/x64アプリケーションをエミュレーションで動作させる仕組みが備わっています。Windows 11 24H2からは新しいエミュレーター「Prism」が導入され、x86命令をArm64命令にJITコンパイルする方式で動作します。通常のアプリケーション（ブラウザやエディタなど）はエミュレーション経由でも大きな問題なく動作することが多いです 💻

https://learn.microsoft.com/en-us/windows/arm/apps-on-arm-x86-emulation

ただし、Microsoft Learnでは「emulation only supports user mode code and doesn't support drivers. Any kernel mode components must be compiled as Arm64（エミュレーションはユーザーモードコードのみをサポートし、ドライバーはサポートしません。カーネルモードコンポーネントはすべてArm64としてコンパイルする必要があります）」と明記されています 🪟

しかし、**かな漢字変換ソフト（IME）** はOSのテキスト入力サブシステムと密接に統合される性質を持っています。WindowsのIMEは**TSF（Text Services Framework）** を介してOSと通信する仕組みで、アプリが起動するとTSFがIMEのDLLをアプリプロセスにロードします 🔧 こうしたシステム密着型のソフトウェアは、一般的にエミュレーション環境での動作保証が難しく、実際にATOKも今回のアップデートまではArm版Windows 11をサポート対象外としていました。

https://learn.microsoft.com/en-us/windows/apps/develop/input/input-method-editor-requirements

ATOKの公式アナウンスでも、今回の対応によって「Armベースプロセッサの性能が発揮されるネイティブアプリ上でのATOKによる入力体験を提供」すると述べられており、ネイティブ動作ならではの恩恵があることが示唆されています ⚡

### ドライバーレベルのソフトウェアが抱える問題

かな漢字変換ソフトは、一般的なユーザーランドアプリケーションとは異なり、システムに深く組み込まれるソフトウェアです。Microsoft Learnの「Windows on Arm FAQ」では、ドライバーについて「kernel-mode drivers and user-mode print drivers MUST be built as native Arm64 binaries to work on Arm64 devices（カーネルモードドライバーおよびユーザーモードプリントドライバーは、Arm64デバイスで動作するためにネイティブArm64バイナリとしてビルドしなければならない）」と記載されています 📘

https://learn.microsoft.com/en-us/windows/arm/faq

また「Troubleshooting x86 desktop apps」では「All kernel-mode drivers, UMDF drivers, and print drivers must be compiled to match the architecture of the OS（すべてのカーネルモードドライバー、UMDFドライバー、プリントドライバーはOSのアーキテクチャに合わせてコンパイルされなければならない）」とも明記されています 📋

https://learn.microsoft.com/en-us/windows/arm/apps-on-arm-troubleshooting-x86

実際、Microsoft Learnの同ページ（Shell extensions）では「Apps that try to hook Windows components or load their DLLs into Windows processes need to recompile those DLLs to match the architecture of the system, such as Arm64. Typically, input method editors (IMEs), assistive technologies, and shell extension apps use these DLLs（Windowsコンポーネントにフックしたり、WindowsプロセスにDLLをロードするアプリは、Arm64などシステムのアーキテクチャに合わせてDLLを再コンパイルする必要があります。通常、IME、支援技術、シェル拡張アプリがこれらのDLLを使用します）」と記載されており、**IMEが明示的にArm64再コンパイルの対象として挙げられています** 📌 これはATOKに限った話ではなく、同様にシステムと密接に関わるソフトウェア全般（セキュリティソフト、VPNクライアントなど）にも共通する課題です。

## Copilot+ PCとArm64アーキテクチャの魅力

### Snapdragon X Eliteが切り拓く新しいPC体験

Qualcomm Snapdragon X Eliteを搭載したCopilot+ PCは、Arm64アーキテクチャの強みを最大限に活かしたデバイスです。Armアーキテクチャ由来の高い電力効率により長時間のバッテリー駆動が可能であり、省電力という観点だけでも選ぶ価値があります 🔋 Copilot+ PCの要件として40 TOPS以上のNPUが必須とされており、Snapdragon XシリーズではQualcomm Hexagon NPUが45 TOPSの性能を提供します 🧠 パフォーマンス面でも、ネイティブARM64アプリではx86エミュレーションと比較して大幅な性能向上が見込めます 🚀

https://learn.microsoft.com/en-us/windows/ai/npu-devices/

Copilot+ PCの特徴として、RecallやLive CaptionsなどMicrosoftの新しいAI機能は2024年の発売当初はSnapdragon Xシリーズ搭載機から提供が始まりました。現在ではIntelやAMDのCopilot+ PCでも利用可能ですが、ArmベースのSnapdragonがこの新カテゴリを牽引してきた存在であることは間違いありません 🤖 こうした背景から、Arm64ネイティブ対応のソフトウェアが充実していくことは、エコシステム全体の価値を高める重要な要素です。

### ネイティブ対応による恩恵

Microsoft Learnの「Windows on Arm FAQ」では、エミュレーションについて「there is performance overhead that comes with emulation. Treat your customers to an Arm-native version of your Windows apps for the best performance, responsiveness, and optimized battery life on their Arm device（エミュレーションにはパフォーマンスのオーバーヘッドが伴います。最高のパフォーマンス、応答性、最適化されたバッテリー寿命のために、Armネイティブ版を提供してください）」と述べられています ⚡

https://learn.microsoft.com/en-us/windows/arm/faq

ATOKがARM64ネイティブで動作することで、このオーバーヘッドがなくなり、キー入力から変換候補表示までの応答性が向上します。日本語入力は一日の作業を通じて常に使い続けるものですから、このレスポンスの改善は体感として大きいはずです 💬 また、ネイティブ実行によりCPU使用率が低下するため、バッテリー持続時間にも好影響を与えます 🔋 そして、ARM64ネイティブで動作するエディタやブラウザ上でも安定した日本語入力ができるようになり、アプリケーション間の親和性が確保されます 🔗

## ATOKのArm64対応の詳細

公式発表によると、今回のアップデートでATOKはArm版Windows 11上のあらゆるアプリケーション形態をサポートしています。ARM64ネイティブアプリ上での日本語入力はもちろん、ARM64（x64互換）モードで動作するアプリ、さらには従来のx64/x86アプリ上での入力にも対応しました ✅

つまり、どのようなアーキテクチャのアプリケーションを使っていても、ATOKが一貫して日本語入力を提供できるようになったわけです。Snapdragon Xシリーズプロセッサーに対応し、Armベースプロセッサの性能がフルに発揮されるネイティブアプリ上でのATOK入力体験が実現されています。

## 同時に追加された注目の新機能

今回のアップデートはArm64対応だけではありません。併せて追加された主な機能も紹介します。

### ATOK MiRA（エイトックミラ）

**ATOK My Intelligent Rewrite Assistant** の略で、生成AIを活用した文章作成アシスタントです。日本語入力の延長線上でシームレスに生成AIと対話し、文章を書き換えることができます 📝

使い方は大きく分けて二通りあり、入力中の文章からそのまま書き換える方法と、入力済みの文章を選択して書き換える方法が用意されています ✏️ 特に注目すべきは「わたしらしく」機能で、ATOKに蓄積されたユーザーの入力傾向（句読点の設定、一文あたりの文字数、漢字の割合など）を生成AIへのプロンプトに自動的に反映することで、画一的ではない「その人らしい」文章が生成されます 🧑‍💻

### ATOK Sync One

2026年6月提供予定の新クラウド同期サービスです。Windows、macOS、Android、iOSの4つのOS間で、変換辞書の学習情報・確定履歴・登録単語などを双方向に同期できるようになります。

## Arm版Windowsエコシステムの成熟

ATOKのArm64対応は、Arm版Windowsにおけるソフトウェア対応の進捗を象徴する出来事です。

ユーザーモードのアプリケーションは比較的早期からエミュレーション経由で動作していましたが、IMEのようなシステム密着型ソフトウェアの対応は時間がかかっていました。今回の対応により、日本語ユーザーがCopilot+ PCを「メインマシン」として使う上でのハードルがまた一つ下がったと言えます。

今後も、VPNクライアント、セキュリティソフト、各種ドライバーなど、システムレベルのソフトウェアのARM64対応が進むことで、Arm版Windowsの実用性はさらに高まっていくでしょう。

## まとめ

2026年2月2日のATOK for Windows（Tech Ver.36）アップデートにより、ATOKはSnapdragon Xシリーズプロセッサー上でARM64ネイティブ動作するようになりました。対応範囲はARM64ネイティブアプリ、ARM64（x64互換）アプリ、x64/x86アプリのすべてに及びます 🎯 同時に、生成AIを活用した文章作成アシスタント「ATOK MiRA」の追加や、2026年6月提供予定の「ATOK Sync One」など、日本語入力の未来を見据えた機能強化も行われています。

Arm版Windows上での日本語入力環境が整備されたことは、Copilot+ PCを日常的に使う日本語ユーザーにとって大きな前進です。省電力かつ高性能なArm64アーキテクチャの恩恵を、日本語入力の場面でもフルに享受できるようになりました 🎉
