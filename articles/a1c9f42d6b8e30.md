## はじめに

GitHub Copilot と聞くと、まず **「コードを自動補完してくれる AI」** というイメージを持つ方が多いのではないでしょうか。

もちろん、それは GitHub Copilot の重要な入口です。エディターでコードを書いているときに次の 1 行や関数全体を提案してくれる体験は、今でも非常に強力です。

しかし、現在の GitHub Copilot はそれだけではありません。計画を立てる、コードを書く、テストする、レビューする、ドキュメントを書く、デプロイ前の確認をする。こうした **開発ワークフローのあらゆる場面で開発者を支える AI パートナー** になりつつあります。

本記事では、「そもそも GitHub Copilot とは何か？」から出発して、開発の流れに沿って GitHub Copilot の位置づけを整理していきます🧭

## 本記事のゴール

対象読者は、GitHub Copilot をコード補完中心に使っている方、またはエージェント機能やクラウド委譲の全体像をまだ整理できていない方です。

- GitHub Copilot を「コード補完ツール」だけでなく、開発ワークフロー全体の支援ツールとして捉え直す
- 計画・実装・テスト・レビュー・デプロイの各場面で、GitHub Copilot が何を助けるのかを理解する
- VS Code 上の GitHub Copilot を中心に、エージェント・カスタマイズ・クラウド委譲の全体像を掴む

## GitHub Copilot とは？

GitHub Copilot は、GitHub が提供する AI コーディングアシスタントです。公式ドキュメントでは、開発者がより少ない労力でコードを書き、問題解決やコラボレーションにより多くの力を使えるようにする存在として説明されています。

GitHub Copilot でできることは、たとえば次のように広がっています。

| 領域 | できること |
|------|------------|
| ✍️ コーディング | 入力中のコード補完、関数やテストコードの生成 |
| 💬 チャット | コードの説明、修正案の相談、設計方針の壁打ち |
| 🧭 計画 | 実装前の調査、タスク分解、実装計画の作成 |
| 🛠️ エージェント | ファイル編集、コマンド実行、テスト実行、自己修正 |
| 🔍 レビュー | Pull Request の説明生成、差分の理解、改善点の相談 |
| ☁️ 委譲 | GitHub 上でバックグラウンド作業を任せ、ブランチや Pull Request にまとめる |

公式ドキュメントでも、Copilot は IDE、GitHub.com、コマンドライン、GitHub Mobile など複数の場所で利用できると説明されています。

参考:

- [What is GitHub Copilot?](https://docs.github.com/en/copilot/about-github-copilot/what-is-github-copilot)
- [GitHub Copilot Chat](https://docs.github.com/en/copilot/using-github-copilot/copilot-chat)

つまり GitHub Copilot は、単一のコード補完機能に限らず、IDE、GitHub.com、CLI、GitHub Mobile など複数の場所で利用できる **AI コーディングアシスタント** として見ると理解しやすいです。

## 「コード補完」から「開発エージェント」へ

従来の GitHub Copilot の分かりやすい価値は、エディター内のコード補完でした。

たとえばコメントを書いた直後に関数の実装候補が出たり、既存コードの文脈から次に書くべき処理を提案してくれたりします。これは今でも日常的に便利です。

一方で、最近の GitHub Copilot は **「次の 1 行を提案する AI」から「目的を受け取って作業を進める AI」** へと役割が広がっています。

VS Code の公式ドキュメントでは、エージェントを「目標を受け取り、手順に分解し、プロジェクト内のファイルを編集し、コマンドを実行し、問題が起きたら自己修正する AI アシスタント」と説明しています。

参考:

- [GitHub Copilot in VS Code](https://code.visualstudio.com/docs/copilot/overview)

この変化はとても大きいです。

コード補完は「今、私が書いているコード」を助けます。エージェントは「これから達成したい開発タスク」を助けます。対象が 1 行からタスク全体へ広がっているわけです。

では、この変化が VS Code 上ではどのような形で現れているのかを整理します。

ここで重要なのは、GitHub Copilot が **「コードを書く瞬間」だけに閉じていない** という点です。VS Code の中に深く統合され、開発の前後工程も含めて支援する方向に進化しています。

## 開発ワークフローで見る GitHub Copilot の役割

ここからは、開発の流れに沿って GitHub Copilot の位置づけを見ていきます。

### 1. プラン：いきなり書かずに、まず考える 🧭

開発で一番危ないのは、要件が曖昧なままコードを書き始めてしまうことです。

GitHub Copilot は、実装前の壁打ち相手として使えます。

- この機能を実装するには、どのファイルを見ればよいか
- 既存の設計に合わせるなら、どの層に処理を置くべきか
- 影響範囲はどこまで広がるか
- 先にテストを書くべき観点は何か
- どの順番で進めると安全か

VS Code では Plan エージェントを使って、コードを書く前に実装計画を作ることもできます。作成した計画を実装に渡すことで、いきなり大きな変更を入れるよりも安全に進めやすくなります。

:::message
Copilot に依頼するときは、「この機能を作って」だけでなく、「まず既存コードを調査して、実装計画を作って」と伝えると、作業の粒度が安定しやすくなります。
:::

### 2. ビルド：実装を一緒に進める ✍️

実装フェーズでは、GitHub Copilot の得意領域が最も分かりやすく現れます。

小さな単位では、インライン補完やインラインチャットが便利です。関数の一部を補完したり、選択範囲に対して「この処理を非同期化して」「この重複をまとめて」と依頼したりできます。

もう少し大きな単位では、エージェントモードが役立ちます。

- 新しい API エンドポイントを追加する
- UI コンポーネントを作成する
- 設定ファイルを追加する
- 既存のサービス層に処理を追加する
- ビルドエラーを見ながら修正する

エージェントはファイル編集だけでなく、必要に応じてコマンドも実行できます。つまり「コードを書いて終わり」ではなく、実際に動くところまで近づけるよう支援できます。

### 3. テスト：品質確認を後回しにしない 🧪

GitHub Copilot は、テスト作成やテスト失敗の調査にも使えます。

たとえば次のような依頼ができます。

- このクラスのユニットテストを追加して
- 異常系のテストケースを洗い出して
- 失敗しているテストの原因を調べて
- 境界値の観点を追加して
- モックの使い方を既存テストに合わせて

テストを書くときに大事なのは、単に「テストコードを生成する」ことではありません。**何を保証したいのかを明確にすること**です。

Copilot は、既存のテストコードを読んでパターンを合わせたり、失敗ログから原因をたどったりする作業を補助できます。開発者は、テストの意図や業務ルールが正しく表現されているかをレビューする役割に集中しやすくなります。

### 4. レビュー：差分を読み、意図を説明する 🔍

レビューは、GitHub Copilot の使いどころが多い領域です。

Pull Request を作る前に、次のような観点で相談できます。

- この差分の要約を作って
- 影響範囲を整理して
- レビューで見てほしいポイントをまとめて
- セキュリティや例外処理の観点で気になる点を探して
- 変更内容に対してドキュメント更新が必要か確認して

GitHub Copilot は Pull Request の説明生成にも使えます。変更の背景、主な修正点、確認したことを整理してくれるため、レビューする側の負担も下げられます。

:::message alert
Copilot が生成したレビュー観点や Pull Request 説明は、そのまま投稿する前に必ず差分と意図を確認してください。
:::

AI が見つけやすいのは、差分上の不整合や一般的なバグの兆候です。一方で、事業上の意図、顧客との約束、チーム固有の設計判断は人間が確認する必要があります。

### 5. デプロイ：出荷前の確認を支える 🚀

デプロイそのものを GitHub Copilot が魔法のように安全にしてくれるわけではありません。

しかし、デプロイ前後の確認作業には Copilot を活用できます。

- リリースノートの下書きを作る
- 変更点から確認観点を洗い出す
- CI の失敗ログを読む
- デプロイ手順の抜け漏れを確認する
- ロールバック手順をドキュメント化する
- 運用担当者向けの説明文を作る

デプロイは「コードをサーバーへ送る作業」ではなく、変更を安全にユーザーへ届ける作業です。Copilot は、その準備と確認を支援する存在として使えます。

## ワークフローを支える Copilot の使い方

ここまで見てきた計画、実装、テスト、レビュー、デプロイの流れを支えるために、Copilot にはいくつかの横断的な使い方があります。

### カスタマイズ：チームの文脈を渡す 🛠️

GitHub Copilot を本当に開発ワークフローに組み込むなら、カスタマイズが重要になります。

何も設定しない状態でも Copilot は使えます。しかし、チームの設計方針や命名規則、テスト方針、禁止事項を知らないままでは、どうしても「それっぽいけれど少し違う」提案になりがちです。

そこで役立つのが、次のような仕組みです。

| 仕組み | 役割 |
|--------|------|
| 📜 Custom instructions | コーディング規約や設計方針を常に伝える |
| 📄 AGENTS.md | エージェントに読ませるリポジトリ固有の作業ルールを書く |
| 🧰 Agent Skills | 特定タスクの手順やテンプレートをまとめる |
| 🤖 Custom agents | レビュー担当、テスト担当、ドキュメント担当のように役割を分ける |
| 🔌 MCP servers | 外部ツールや社内システムの情報を Copilot から使えるようにする |
| 🪝 Hooks（Preview） | 特定イベントでコマンドを実行し、検証や自動化につなげる |

VS Code の公式ドキュメントでも、Custom instructions、Agent Skills、Custom agents、MCP servers、Hooks などを使って AI をワークフローに合わせる方法が紹介されています。

参考:

- [Customize AI for your workflow - GitHub Copilot in VS Code](https://code.visualstudio.com/docs/copilot/overview#_customize-ai-for-your-workflow)

Copilot を「賢い外部の人」として扱うのではなく、**チームの文脈を知っている共同作業者に近づける**。ここがカスタマイズの目的です。

### 管理：複数の作業を見渡す 👀

開発では、ひとつのタスクだけを進めているとは限りません。

バグ修正、リファクタリング、テスト追加、ドキュメント更新、調査タスク。複数の作業が並行して動くことはよくあります。

VS Code の Copilot では、**1 セッション = 1 エージェントの作業単位** という設計になっています。Sessions view でセッションを管理することがそのままエージェント管理になっており、進行中・未読のバッジで状態を確認しながら、セッションを切り替えて作業を進められます。

また、エージェントが複雑なタスクを処理するとき、内部でサブエージェントを呼び出すことがあります。サブエージェントはチャット内に折り畳み可能なツール呼び出しとして表示されるため、何が行われているかを追いかけることができます。ただし、サブエージェントの起動はエージェントが自律的に判断するものであり、ユーザーが直接操作する独立した管理 UI はありません。

さらに、セッションタイプとして **Copilot CLI セッション** を選ぶと、エージェントがローカルマシンのバックグラウンドで独立して動作します。ローカルエージェントと異なり、VS Code を閉じてもセッションは継続します。変更の適用方法も選べ、**Worktree 分離**（Git worktree を自動作成し、作業ブランチを汚さない）と **Workspace 分離**（現在のワークスペースに直接適用）の 2 モードがあります。複数の Copilot CLI セッションを並行して走らせることも可能で、独立したタスクを同時に進められます。

ローカルエージェントで要件を詰めてから Copilot CLI に引き渡す「ハンドオフ」機能もあり、会話履歴とコンテキストをそのまま引き継いでバックグラウンドで実行させることができます。

これは、開発者が AI に作業を投げっぱなしにするための仕組みではありません。むしろ、**どのタスクがどこまで進んでいるかを可視化し、人間が判断しやすくするための仕組み**です。

参考:

- [Manage chat sessions - VS Code](https://code.visualstudio.com/docs/copilot/chat/chat-sessions)
- [Copilot CLI sessions in Visual Studio Code](https://code.visualstudio.com/docs/copilot/agents/copilot-cli)
- [Subagents in Visual Studio Code](https://code.visualstudio.com/docs/copilot/agents/subagents)

### 委譲：バックグラウンドやクラウドに任せる ☁️

GitHub Copilot は、ローカルの VS Code だけでなく、GitHub 上のクラウドエージェントとしても利用できます。

公式ドキュメントでは、Copilot cloud agent はリポジトリを調査し、実装計画を作り、ブランチ上でコードを変更し、必要に応じて Pull Request を作成できると説明されています。

参考:

- [About GitHub Copilot cloud agent](https://docs.github.com/en/copilot/concepts/agents/cloud-agent/about-cloud-agent)

この機能の位置づけは、ペアプログラミングというより **非同期の開発委譲** に近いです。

たとえば、次のようなタスクは委譲しやすいです。

- 小さな不具合修正
- テスト追加
- ドキュメント更新
- 既存パターンに沿ったリファクタリング
- 影響範囲が限定された改善

公式ドキュメントでも、クラウドエージェントはスコープと必要な文脈が明確なタスクに適していると説明されています。そのため、要件が曖昧な変更やリスクの高い変更では、人間が強く関与したほうが安全です。

Issue や Pull Request の会話、リポジトリ、カスタム instructions、セッション内で実行したテストや lint の結果など、参照できる文脈が揃っているほど、Copilot に具体的な依頼をしやすくなります。

### モダナイズ：古いコードと向き合う 🧹

現実の開発では、新規開発よりも既存コードの改善に多くの時間を使うことがあります。

古いフレームワーク、読みにくくなった設計、テスト不足、属人化した処理。こうしたコードに向き合うときも、GitHub Copilot は役立ちます。

- 既存コードの構造を説明してもらう
- 依存関係を整理する
- リファクタリング計画を作る
- 小さな単位に分けて変更する
- 既存仕様を保護するテストを追加する

モダナイズで重要なのは、一気に作り直さないことです。

Copilot にも「一括で全部直して」と頼むのではなく、「まず構造を調べる」「安全な分割方針を出す」「テストで保護する」「小さく変更する」という順番で依頼すると、現実的に進めやすくなります。

## Copilot を使うときの考え方

GitHub Copilot を開発ワークフローに組み込むとき、私は次のように考えると扱いやすいと感じています。

| 考え方 | 説明 |
|--------|------|
| 🧭 目的を伝える | 「何をしたいか」「なぜ必要か」を書く |
| 📦 文脈を渡す | 関連ファイル、制約、既存ルールを伝える |
| 🔁 小さく回す | 大きな変更を一度に任せず、計画・実装・確認を分ける |
| 🧪 検証を含める | テストやビルドの実行まで依頼する |
| 👀 最後は人間が見る | 仕様、セキュリティ、リリース判断は人間が責任を持つ |

Copilot は「何でも自動で正解にしてくれる存在」ではありません。

むしろ、開発者が目的と制約を言語化するほど、より良い結果を返してくれる存在です。これは少し面倒に見えますが、実はチーム開発においても大切な作業です。

AI に分かるように説明できるタスクは、人間にも共有しやすいタスクになります。

## まとめ

GitHub Copilot は、もはや単なるコード補完ツールではありません。

もちろん、コードを書く瞬間の支援は今でも重要です。しかし現在の Copilot は、計画、実装、テスト、レビュー、管理、委譲、デプロイ前の確認、モダナイズまで、開発ワークフロー全体に関わる存在になっています。

一部 Preview の機能を含みますが、VS Code では Plan エージェント、Custom instructions、Agent Skills、Custom agents、MCP servers、Hooks などを使って、Copilot を開発スタイルに合わせてカスタマイズできます。

大切なのは、Copilot を **「コードを書かせる道具」ではなく「開発の流れを一緒に進めるパートナー」** として捉えることです。

計画を一緒に作り、実装を進め、テストで確かめ、レビューで磨き、リリースに向けて整える。そう考えると、GitHub Copilot の使いどころは一気に広がります。

これからの開発では、「どのコードを書くか」だけでなく、**どの作業を人間が担い、どの作業を AI と協働するか** を設計することが、開発者にとってますます重要になっていきそうです🤖
