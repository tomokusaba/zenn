## はじめに

2026年2月18日、Microsoft Fluent UI Blazorライブラリの **v5.0 RC1（Release Candidate 1）** が公開されました。このメジャーリリースは、ライブラリの基盤そのものを刷新する大規模なアップデートであり、最新のFluent UI Web Components 3の採用、宣言的なレイアウトシステム、グローバルデフォルト値機構、そしてローカライゼーションサポートなど、数多くの新機能を含んでいます。

本記事では、現行のFluent UI Blazor v4（バージョン4.14.0）との比較を交えながら、v5 RC1で何が変わったのかを詳しく解説します。

## Fluent UI Blazorとは

Fluent UI Blazorは、MicrosoftのFluent Design Systemに準拠したRazorコンポーネントライブラリです。Blazorアプリケーションに対して、Microsoft 365やTeamsと同様のモダンなUI体験を提供することを目的としています。一部のコンポーネントはFluent UI Web Componentsのラッパーとして実装されており、その他はFluent Design Systemを活用した独自のコンポーネントとして提供されています。

NuGetパッケージ名は `Microsoft.FluentUI.AspNetCore.Components` で、.NET 8以降をサポートしています。オープンソースプロジェクトとしてGitHubの [microsoft/fluentui-blazor](https://github.com/microsoft/fluentui-blazor) リポジトリで開発が進められています。

## v5の根幹的変化：Fluent UI Web Components 3への移行

v5における最も重要な変化は、**レンダリング基盤の完全な刷新**です。

v4では、FAST（Microsoft製のWeb Componentsフレームワーク）をベースとしたWeb Componentsを利用していました。具体的には、`@microsoft/fast-element` v1.13.0と `@microsoft/fast-foundation` v2.49.6に依存しており、v4の公式ドキュメントでは「コンポーネントはFASTのAdaptive UI技術上に構築されており、Design Tokensを通じてデザインのカスタマイズとパーソナライゼーションを実現し、アクセシビリティを自動的に維持する」と説明されています。`FluentDesignTheme` コンポーネントや各種Design Tokensを通じて、テーマのダーク/ライト切替やカラーカスタマイズを行う構成でした。

v5では、この基盤が **Fluent UI Web Components v3** に置き換えられています。これはMicrosoft 365、Teams、Windows 11といったMicrosoft自身の製品で使用されているものと同一のコンポーネント群です。この移行により、以下の恩恵が得られます。

**Fluent 2デザインとの完全な一致** — Blazorコンポーネントが、Microsoftの製品と寸分違わないルック＆フィールでレンダリングされるようになりました。v4では微妙な視覚的差異が存在する場合がありましたが、v5ではその差異が解消されています。

**パフォーマンスの向上** — 新しいWeb Componentsはより軽量かつ効率的に設計されており、JavaScriptのフットプリントが削減されたことで、初回ロード時間の短縮とよりスムーズなインタラクションが実現されています。

**アクセシビリティの改善** — 新たなWeb Components層ではアクセシビリティがファーストクラスの設計原則として組み込まれており、WCAG 2.1 AAへの準拠が標準で提供されます。

**バンドルサイズの削減** — レガシーなFAST依存を除去したことで、パッケージ全体のサイズが大幅に削減されました。

## FASTプロジェクトの現在

Fluent UI Blazor v5がFASTベースの基盤から離脱したことを受けて、「FASTプロジェクト自体はどうなっているのか」「v4が使っていたFASTパッケージはまだメンテナンスされているのか」という疑問を持つ方もいるかもしれません。正確に理解するには、FASTエコシステム内の各パッケージの役割と現在のステータスを区別する必要があります。

### Fluent UI Blazor v4が実際に依存していたFASTパッケージ

Fluent UI Blazor v4.14.0のソースコード（[`src/Core.Assets/package.json`](https://github.com/microsoft/fluentui-blazor/blob/v4.14.0/src/Core.Assets/package.json)）を確認すると、v4は以下の2つのFASTパッケージに直接依存していました。

- **`@microsoft/fast-element` v1.13.0** — Web Componentsを定義するためのコアエンジン（テンプレート、リアクティブバインディング、Shadow DOM管理など）
- **`@microsoft/fast-foundation` v2.49.6** — `fast-element` 上に構築されたスタイルなしの基盤コンポーネント群（ボタン、ダイアログ、ツリービューなど、デザインシステム非依存のUIパーツ）

v4は、これらの基盤コンポーネントにFluent Designのスタイル（Design Tokens）を適用し、Blazorのコンポーネントモデルでラップするというアーキテクチャでした。

### v4が使っていたFASTパッケージの現在のステータス

これらのパッケージに関して言えば、**事実上の開発凍結状態**にあります。

`@microsoft/fast-foundation` は、最後の安定版リリースがv2.50.0（約1年前に公開）であり、それ以降新しいバージョンは公開されていません。かつてv3.0.0-alphaシリーズの開発も試みられていましたが、alpha.33（約2年前）を最後に停止しており、安定版としてリリースされることはありませんでした。npmでは依然として約170K/週のダウンロード数を記録していますが、これは既存プロジェクトの `npm install` による推移的依存解決が主な要因と考えられます。

`@microsoft/fast-element` のv1.x系統についても同様で、最後のv1.xリリースはv1.14.0（約1年前）です。重要なのは、`fast-element` のv2.x系統（後述）はv1.xとはAPIが大きく異なる別系統であり、v1.xからのアップグレードパスは提供されていないという点です。

つまり、**Fluent UI Blazor v4が依存していたFASTの「基盤コンポーネント層」（`fast-foundation`）と「コアエンジンv1.x」（`fast-element` v1系）は、いずれも約1年以上新しいリリースが行われていない状態**です。これは、FASTプロジェクト全体としてコンポーネント基盤層（`fast-foundation`）から、より低レベルなコアエンジン層（`fast-element` v2.x）へと開発リソースをシフトさせたことを反映しています。

### FASTの位置づけの変化とMicrosoftのコンポーネント戦略

Fluent UI Blazorにおける「FAST離れ」は、FASTの品質に問題があったからではなく、Microsoft内部のコンポーネント戦略の転換が背景にあります。

v4時代の構造は、**FAST Foundation → Fluent UI Web Components v2.x → Fluent UI Blazor** という依存チェーンでした。`@fluentui/web-components` v2.6.1（最後の安定版、約2年前に公開）は、パッケージの説明文に「`@microsoft/fast-foundation` をcomposeし、MicrosoftのFluent Design言語をサポートする」と明記されていたように、FAST基盤の上に構築されたものでした。

v5では、この構造が **Fluent UI Web Components v3 → Fluent UI Blazor** に置き換わっています。Fluent UI Web Components v3（`@fluentui/web-components` v3.0.0-rc.6）は、`microsoft/fluentui` モノレポ内で開発される完全に新しいコンポーネント群です。`@fluentui/tokens` と `setTheme` APIを基盤としており、FAST Foundationへの依存は一切ありません。このコンポーネント群はMicrosoft 365、Teams、Windows 11で実際に使用されているものと同一であり、Fluent UI Reactコンポーネントとの属性名の整合性も取られています。Blazorライブラリがこの新しいコンポーネント群に切り替えたのは、Microsoftのプロダクト全体でのデザイン一貫性を実現するための戦略的判断です。

### FAST Elementの現在と将来：v2.xからv4.0へ

FASTプロジェクト全体が停止しているわけではありません。**コアエンジンである `@microsoft/fast-element` v2.x系統は活発に開発が続けられています**。v2.10.0が2026年2月に公開されており、npmでの週間ダウンロード数は約177Kに達しています。ただし、このv2.x系統はv4が使っていたv1.x系統とは異なるAPIを持つ別系統であり、`fast-foundation` のような「すぐに使えるUIコンポーネント基盤」ではなく、Web Componentsを構築するための「低レベルなリアクティブエンジン」としての位置づけです。

FASTプロジェクトのGitHubリポジトリ（[microsoft/fast](https://github.com/microsoft/fast)）では、2025年11月に **`@microsoft/fast-element` 4.0.0のメジャーリリース提案**（[Issue #7210](https://github.com/microsoft/fast/issues/7210)）が公開されました。この提案には、宣言的テンプレート統合（`@microsoft/fast-html` パッケージの `@microsoft/fast-element` への統合）、Declarative Shadow DOMの活用強化、ハイドレーションの改善、非同期メソッドの導入、Scoped Custom Element Registriesの探求、SVGテンプレーティングの改善など、多岐にわたる変更が含まれています。

ロードマップ（[FAST Architecture Roadmap](https://github.com/orgs/microsoft/projects/240/views/1)）には、「**FAST Element vNext**」（パフォーマンス最適化、パッケージサイズ削減）、「**SSR 2.0**」（サーバーサイドレンダリングとハイドレーション改善）、「**FAST Element Future**」（4.0.0に向けた計画）といった作業カテゴリが設定されています。GitHubリポジトリ自体もアクティブに更新されており、155名のコントリビューターと9,600以上のスターを持つプロジェクトとして、Web Componentsエコシステムにおいて依然として重要な存在です。

### Fluent UI BlazorユーザーへのFASTの意義

Fluent UI Blazor v5ではFAST依存が完全に除去されたため、Blazor開発者がFASTを直接意識する必要はなくなりました。v4で利用していた `fast-foundation` のDesign TokensやAdaptive UIの概念は、v5のFluent UI Web Components v3が提供する `@fluentui/tokens` ベースのテーマ機構に置き換えられています。

一方、FASTの `@microsoft/fast-element` v2.xは、W3C Web Components標準に準拠した軽量ライブラリ（ツリーシェイキング前で約10KB、最適化後は約4.5KB）として、フレームワークに依存しないカスタムWeb Componentsの構築に引き続き活用できます。Blazor以外のフレームワーク（Angular、React、Vueなど）との統合も可能であり、独自のデザインシステムをWeb Components標準で構築したい場合には、依然として有力な選択肢です。ただし、v4のように「FASTの上にFluent UIコンポーネントが乗る」という構造は、Microsoftのエコシステムにおいては過去のものとなりました。

## セットアップ方法の比較：v4 vs v5

### v4のセットアップ

v4では、NuGetパッケージのインストール後に以下のような設定が必要でした。

```bash
dotnet add package Microsoft.FluentUI.AspNetCore.Components
```

`Program.cs` でサービスを登録します。

```csharp
builder.Services.AddFluentUIComponents();
```

そして `MainLayout.razor` の末尾に、用途別の**個別プロバイダーコンポーネント**を追加する必要がありました。

```razor
<FluentToastProvider />
<FluentDialogProvider />
<FluentTooltipProvider />
<FluentMessageBarProvider />
<FluentMenuProvider />
```

Web Componentsのスクリプトはライブラリに含まれており、Blazorの仕組みにより自動ロードされていました。ただし、.NET 8のSSR（Static Server Rendering）モードを使用する場合には `App.razor` でスクリプト参照を手動で追加する必要がありました。また、テンプレートを使用しない場合は `Reboot.css` の手動インクルードが必要で、TailwindなどのCSSフレームワークと併用する際にはクラス名バリデーションの無効化設定も求められました。

### v5のセットアップ

v5ではセットアップが大幅にシンプルになりました。

```bash
dotnet add package Microsoft.FluentUI.AspNetCore.Components --prerelease
dotnet add package Microsoft.FluentUI.AspNetCore.Components.Icons
```

`_Imports.razor` に名前空間を追加します。

```razor
@using Microsoft.FluentUI.AspNetCore.Components
@using Icons = Microsoft.FluentUI.AspNetCore.Components.Icons
```

`Program.cs` でサービスを登録します。

```csharp
builder.Services.AddFluentUIComponents();
```

そして `MainLayout.razor` の末尾にプロバイダーコンポーネントを配置します。

```razor
@* Add all FluentUI Blazor Providers *@
<FluentProviders />
```

v4では `FluentToastProvider`、`FluentDialogProvider`、`FluentTooltipProvider`、`FluentMessageBarProvider`、`FluentMenuProvider` の5つを個別に配置する必要がありましたが、v5では **`<FluentProviders />`** 一つに統合されました。

さらに注目すべき点は、**追加のJavaScriptやCSSファイルの参照が一切不要**であることです。Web Componentのスクリプトはライブラリによって自動的にインクルードされ、ロードされます。v4のSSR環境で必要だったスクリプトの手動追加や、`Reboot.css` の手動設定といった追加手順が排除されています。

## 新機能1：FluentLayoutコンポーネント

v5では、アプリケーションの構造的な骨格となる **`FluentLayout`** コンポーネントが完全に再設計されました。CSS GridやFlexboxを手動で組み立てる代わりに、宣言的なエリアベースのシステムでフルページレイアウトを構築できます。

### v4でのレイアウト構築

v4では、レイアウトの構成は開発者の責任でした。`FluentStack` や標準的なHTMLとCSSを用いて、ヘッダー、ナビゲーション、コンテンツ、フッターの配置を手動で構築する必要がありました。レスポンシブ対応やナビゲーションパネルの折りたたみ動作も、開発者自身が実装する必要がある場合がほとんどです。

### v5のFluentLayoutによるアプローチ

v5の `FluentLayout` では、`FluentLayoutItem` に `LayoutArea` を指定するだけで、各コンテンツの配置先をフレームワークに伝えることができます。

```razor
@inherits LayoutComponentBase

<FluentLayout>

    <!-- Header -->
    <FluentLayoutItem Area="@LayoutArea.Header">
        <FluentStack VerticalAlignment="VerticalAlignment.Center">
            <FluentLayoutHamburger />
            <FluentText Weight="TextWeight.Bold"
                        Size="TextSize.Size400">
                My Application
            </FluentText>
            <FluentSpacer />
        </FluentStack>
    </FluentLayoutItem>

    <!-- Navigation -->
    <FluentLayoutItem Area="@LayoutArea.Navigation"
                      Width="250px">
        <NavMenu />
    </FluentLayoutItem>

    <!-- Content -->
    <FluentLayoutItem Area="@LayoutArea.Content"
                      Padding="@Padding.All3">
        @Body
    </FluentLayoutItem>

    <!-- Footer -->
    <FluentLayoutItem Area="@LayoutArea.Footer">
        Powered by Microsoft Fluent UI Blazor
    </FluentLayoutItem>

</FluentLayout>

<FluentProviders />
```

`FluentLayoutHamburger` コンポーネントは、ナビゲーションパネルの表示・非表示を切り替えるハンバーガーボタンを自動的にレンダリングします。スムーズなアニメーションとレスポンシブ動作がJavaScript不要で実現されています。

`LayoutArea.Header`、`LayoutArea.Navigation`、`LayoutArea.Content`、`LayoutArea.Footer` という型付きのエリア定義により、コンテンツの配置ミスが防止されます。ナビゲーションパネルの幅やコンテンツ領域のパディングもパラメータで簡潔に制御できるため、アプリケーション全体で一貫したレイアウト構造が維持されます。

## 新機能2：デフォルト値システム

v5では、コミュニティから長く待ち望まれていた**デフォルト値（Default Values）システム**が導入されました。この機構により、任意のFluent UIコンポーネントに対してグローバルなデフォルトパラメータ値を一度だけ定義し、アプリケーション全体に適用できます。

### v4での課題

v4では、すべてのボタンに特定のスタイルを適用したい場合、個々のインスタンスに対して明示的にパラメータを設定する必要がありました。

```razor
<FluentButton Appearance="@Appearance.Accent">Save</FluentButton>
<FluentButton Appearance="@Appearance.Accent">Submit</FluentButton>
<FluentButton Appearance="@Appearance.Accent">Confirm</FluentButton>
```

この方法は手間がかかり、設定忘れが発生しやすく、デザイン変更の際には大量のファイルを修正しなければならないという保守性の問題を抱えていました。

### v5でのソリューション

v5では、`Program.cs` でグローバルなデフォルト値を設定できます。

```csharp
builder.Services.AddFluentUIComponents(config =>
{
    // FluentButtonコンポーネントのデフォルト値を設定
    config.DefaultValues.For<FluentButton>()
          .Set(p => p.Appearance, ButtonAppearance.Primary);
    config.DefaultValues.For<FluentButton>()
          .Set(p => p.Shape, ButtonShape.Circular);
});
```

この設定により、アプリケーション内のすべての `FluentButton` は自動的に `Primary` の外観と `Circular` の形状を使用します。特定のインスタンスで異なる値が必要な場合には、ローカルパラメータで明示的にオーバーライドできます。

```razor
<!-- グローバルデフォルト（Primary + Circular）を使用 -->
<FluentButton>Save</FluentButton>

<!-- このインスタンスだけ外観をオーバーライド -->
<FluentButton Appearance="ButtonAppearance.Outline">Cancel</FluentButton>
```

APIはラムダ式を使用しており、IntelliSenseが完全にサポートされるため、マジックストリングに頼る必要がありません。デザイン上の決定を一か所で管理できることで、大規模アプリケーションや共有デザインシステムにおける一貫性の確保と保守性の向上が実現されます。

## 新機能3：ローカライゼーションサポート

v5には、コンポーネント内部の文字列（ボタンラベル、ARIA属性、アクセシビリティテキストなど）を任意の言語に翻訳するための**ビルトインローカライゼーションシステム**が導入されています。

v4にはこのような統合されたローカライゼーション機構は存在せず、コンポーネント内部の文字列の翻訳は開発者側で個別に対応する必要がありました。

v5では `IFluentLocalizer` インターフェースを実装することでローカライゼーションを実現できます。

```csharp
using Microsoft.FluentUI.AspNetCore.Components;

public class CustomFluentLocalizer : IFluentLocalizer
{
    public string this[string key, params object[] arguments]
    {
        get
        {
            return key switch
            {
                "SomeKey" => "カスタム翻訳テキスト",
                "AnotherKey" => string.Format("別の翻訳 {0}", arguments),

                // 翻訳が見つからない場合は英語にフォールバック
                _ => IFluentLocalizer.GetDefault(key, arguments),
            };
        }
    }
}
```

`Program.cs` で登録します。

```csharp
builder.Services.AddFluentUIComponents(config =>
    config.Localizer = new CustomFluentLocalizer());
```

埋め込みリソース（.resx）ファイルを使用した多言語サポートにも対応しています。各コンポーネントのドキュメントページには、ローカライズ可能な文字列の一覧が記載されています。

## コードの変更点とマイグレーション

v5の公式ブログでは、既存のコンポーネントの大部分についてマイグレーションは比較的スムーズに行えると説明されています。`FluentButton` や `FluentSelect` といったコンポーネント名は引き続き使用でき、内部的に新しいFluent UI Web Componentsでレンダリングされる仕組みに切り替わります。

ただし、新しいWeb ComponentsがFluent UI Reactコンポーネントと属性名を合わせる形で整合性が取られたため、多くのコンポーネントにおいて**significant changesが加えられた**と公式ブログで明記されています。たとえば、v4で `Appearance` 列挙型（`Appearance.Accent` 等）として定義されていた値が、v5では `ButtonAppearance.Primary` のようにコンポーネント固有の型名に変更されています。また、v4で使用していた個別プロバイダー（`FluentToastProvider` など）は `FluentProviders` に統合されました。

公式チームは [マイグレーションガイド](https://fluentui-blazor-v5.azurewebsites.net/MigrationV5) を通じて詳細な移行手順のドキュメントを多数用意しています。さらに、v5ではライブラリ専用の [MCPサーバー](https://fluentui-blazor-v5.azurewebsites.net/Mcp) も公開されており、日常的な開発やマイグレーション作業における支援ツールとして利用できます。

## v4とv5の主要な違いまとめ

| 項目 | v4（4.14.0） | v5（RC1） |
|---|---|---|
| **基盤技術** | FAST Web Components | Fluent UI Web Components v3 |
| **デザイン準拠** | Fluent Design（微差あり） | Fluent 2完全準拠 |
| **レイアウト** | 手動構築（CSS/Flexbox） | `FluentLayout` 宣言的エリアベース |
| **デフォルト値** | 個別設定のみ | グローバルデフォルト値システム |
| **ローカライゼーション** | 開発者が個別対応 | `IFluentLocalizer` ビルトイン対応 |
| **プロバイダー** | 個別5コンポーネント | `FluentProviders` に統合 |
| **JS/CSS参照** | 基本自動（SSR時は手動追加） | 完全自動ロード |
| **アクセシビリティ** | WCAG対応 | WCAG 2.1 AA標準準拠 |
| **バンドルサイズ** | FAST依存あり | FAST依存除去で軽量化 |
| **MCPサーバー** | なし | 開発・マイグレーション支援用に提供 |
| **対応.NETバージョン** | .NET 8 / 9 / 10 | .NET 8 / 9 / 10 |

## 現時点の注意点

v5 RC1はAPIの安定化が進んだ段階ですが、いくつかのコンポーネントがまだ実装中であり、今後のRC版で追加される予定です。未実装コンポーネントの一覧は [dev-v5のTODOリスト](https://github.com/microsoft/fluentui-blazor/issues/4182) で確認できます。

本番環境への適用については、RC版の特性を理解した上で判断してください。問題を発見した場合には、再現可能なコードとともにGitHubにIssueとして報告することが推奨されています。

## まとめ

Fluent UI Blazor v5 RC1は、FASTベースからFluent UI Web Components v3への基盤移行という根本的な変化を軸に、開発者体験を大幅に向上させるリリースです。`FluentLayout` による宣言的レイアウト、デフォルト値システムによるグローバルな設定管理、ビルトインのローカライゼーションサポートは、いずれもv4で開発者が抱えていた課題を直接的に解決するものです。

v4からv5への移行パスも提供されており、MCPサーバーによる開発支援も用意されています。Blazorを用いたMicrosoftスタイルのアプリケーション開発において、v5は重要なマイルストーンとなるでしょう。
