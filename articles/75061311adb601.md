# はじめに

Webアクセシビリティの話になると、私たちはまず「見えるか」「押せるか」「読めるか」を考えます。

もちろんそれらは重要です。ただ、そこで見落としやすいのが、**その情報がブラウザーや支援技術に正しく伝わっているか**という視点です。

WCAG 2.2 の 4 つの原則のうち、最後に置かれている **Robust（堅牢）** は、この点を扱います。W3C はこの原則を、**コンテンツは支援技術を含むさまざまなユーザーエージェントによって解釈できるほど堅牢でなければならない**と説明しています。

「堅牢」という言葉だけ聞くと、サーバーが落ちにくいとか、古いブラウザーでも壊れないとか、そういう意味に聞こえるかもしれません。しかし WCAG の文脈での堅牢は、もう少し具体的です。

それは、**人が見て理解している UI の意味や状態を、ソフトウェア側も同じように解釈できること**です。

この記事では、日常生活のエピソードから「堅牢」を捉え直しつつ、WCAG 2.2 の Guideline 4.1 Compatible、特に 4.1.2 Name, Role, Value と 4.1.3 Status Messages を中心に整理します。

# 改札機の残高表示があやふやになる

駅の自動改札を通るとき、Suica の残高表示が一瞬出ますが、混雑している時間帯だと、あの表示が自分のものなのか、ひとつ前の人のものなのか、タイムラグのせいであやふやに感じることがあります。

たとえば、前の人が通った直後に自分も続けて通過すると、表示された残高が本当に自分の金額なのか、ひとつ前の人の利用結果がまだ残っているだけなのか、一瞬判断に迷うことがあります。改札そのものは通れているのに、表示と自分の行動の結び付きが少し曖昧になるわけです。

もしその改札が、色の変化だけで状態を伝えていたらどうでしょうか。もし短い音だけで通過可否を区別していたらどうでしょうか。もし画面に小さくエラー文言が出ていても、それが読み上げ装置や補助表示に渡っていなければどうでしょうか。

その瞬間に起きる問題は、単に「見えにくい」ではありません。**今表示されている状態が、誰のどの操作に対応しているのかが正しく伝わっていない**のです。

Web の UI でも同じことが起きます。見た目の上では「検索結果が読み込まれました」「保存できました」「入力エラーがあります」と表示していても、その変化が支援技術に伝わらなければ、一部のユーザーにとっては何も起きていないのと同じです。

堅牢とは、この取りこぼしを減らすための原則です。

# WCAG における「堅牢」とは何か

WCAG 2.2 の原則 4 は、次のように定義されています。

> Content must be robust enough that it can be interpreted by a wide variety of user agents, including assistive technologies.

その下にある Guideline 4.1 Compatible は、さらに次のように述べています。

> Maximize compatibility with current and future user agents, including assistive technologies.

ここで言う user agents には、一般的なブラウザーだけでなく、スクリーンリーダー、音声入力、拡大表示、点字ディスプレイなど、さまざまな支援技術が含まれます。

つまり堅牢の本質は、HTML やスクリプトを「それっぽく動かす」ことではなく、**UI の意味と変化を機械可読な形で露出すること**にあります。

W3C の Understanding Guideline 4.1 では、その意図を次の 2 点で説明しています。

1. 支援技術を壊したり回避したりする実装を避けること
2. 支援技術が認識し操作できる標準的な方法で情報を露出すること

この視点に立つと、「見た目は同じだが div で全部作ったカスタムUI」と「標準の button や input を適切に使ったUI」の差が、単なる書き方の好みではないことが見えてきます。

# 4.1.1 は WCAG 2.2 で削除された

堅牢を調べると、昔の情報として 4.1.1 Parsing がよく出てきます。

ただし、WCAG 2.2 では **4.1.1 Parsing は obsolete and removed、つまり廃止・削除** とされています。

これは「構文の正しさがどうでもよくなった」という意味ではありません。W3C は、支援技術が HTML を直接解析していた時代の課題が薄れ、4.1.1 が独立した達成基準としては有用性を失った、と説明しています。

その結果、WCAG 2.2 で堅牢を考えるうえで中心になるのは、次の 2 つです。

- 4.1.2 Name, Role, Value
- 4.1.3 Status Messages

実務でも、ここに集中した方が有益です。つまり、**部品の意味が伝わるか**、そして **状態変化が伝わるか** を確認することです。

# 4.1.2 Name, Role, Value

4.1.2 は、フォーム要素、リンク、スクリプトで生成されたコンポーネントを含む UI 部品について、少なくとも次をプログラムで判定できるよう求めています。

- name: その部品が何であるか
- role: その部品がどんな種類のUIか
- value / state / property: 現在の値や状態がどうなっているか

たとえば、見た目がボタンに見えても、実体がただの div なら、スクリーンリーダーには「押せるもの」として伝わらない可能性があります。

## 悪い例: 見た目だけボタン

```html
<div class="button" onclick="submitForm()">送信</div>
```

これはマウス操作では動くかもしれません。しかし、role がなく、キーボード操作の前提も弱く、支援技術にとっては単なるテキストの塊として扱われることがあります。

## 良い例: 標準要素を使う

```html
<button type="submit">送信</button>
```

標準の button 要素を使えば、名前、役割、操作方法の多くが仕様とユーザーエージェント側で自然に担保されます。

W3C も 4.1.2 の説明で、**標準の HTML コントロールを仕様どおりに使う場合、この達成基準は通常満たしやすい**と述べています。

これは堅牢の実務において非常に重要です。カスタムUIを完全に避ける必要はありませんが、避けられる場面でわざわざ意味情報を自前で再実装しない方が安全です。

## カスタムUIで必要になること

どうしても独自のコンポーネントを作るなら、少なくとも次を明示する必要があります。

```html
<div
	role="button"
	tabindex="0"
	aria-pressed="false"
	onclick="toggleFavorite()"
	onkeydown="handleFavoriteKey(event)">
	お気に入り
</div>
```

この例では次の情報を支援技術に渡しています。

- role="button": これはボタンである
- tabindex="0": キーボードでフォーカスできる
- aria-pressed="false": 押下状態を持つトグルボタンである

ただし、ここまでやっても標準要素より保守は難しくなります。キーボードイベント、フォーカス表示、状態同期、名前付けの一貫性など、責任範囲が一気に広がるからです。

堅牢の観点では、**凝った見た目より、意味が崩れない実装の方が価値が高い**ことが多いです。

# 4.1.3 Status Messages

堅牢を考えるうえで、もう一つ見落とされやすいのが 4.1.3 Status Messages です。

これは、フォーカス移動を伴わない状態メッセージを、role や property によって支援技術が把握できるようにすることを求めます。

身近な例でいえば、券売機やセルフレジが「処理中です」「支払いが完了しました」「バーコードを読み取れませんでした」と知らせる場面に近いです。

利用者は、画面全体が切り替わらなくても、そのメッセージをきっかけに次の行動を決めます。だからこそ、その変化が一部のユーザーにだけ届かない設計は問題になります。

## よくある失敗例

```html
<div id="resultMessage"></div>

<script>
	resultMessage.textContent = "保存しました";
</script>
```

見た目には表示されますが、これだけではスクリーンリーダーが確実に通知してくれるとは限りません。

## 改善例: role や live region を使う

```html
<div id="resultMessage" aria-live="polite"></div>

<script>
	resultMessage.textContent = "保存しました";
</script>
```

あるいは、より意味の強い status role を使う方法もあります。

```html
<div role="status" aria-live="polite"></div>
```

エラーのように優先度が高い通知では、状況によって alert を検討することもあります。

```html
<div role="alert">入力内容に誤りがあります</div>
```

W3C の一次情報でも、成功メッセージや処理結果には role="status"、エラー通知には role="alert" や適切な live region を使う整理になっています。

重要なのは、**画面の中で起きた変化を、視覚だけに閉じ込めないこと**です。

SPA やモーダル中心の画面では、ページ遷移がないぶん、この問題が特に起きやすくなります。ここで注意したいのは、W3C の Understanding によれば、**検索結果の一覧そのもの**は通常 4.1.3 の status message には含まれないことです。一方で、「検索中です」「18 件見つかりました」「保存しました」「入力エラーがあります」といった短い状態通知は 4.1.3 の対象になります。

そのため、Ajax で検索結果を差し替える場合でも、見直すべきなのは結果一覧そのものより、結果件数や読み込み完了、保存成功、エラー発生といった**状態メッセージの出し方**です。

# 「堅牢」は未来の支援技術への保険でもある

Guideline 4.1 は current and future user agents と書いています。

これは今動いている特定のスクリーンリーダーだけを対象に最適化する、という意味ではありません。**標準的な意味づけを守ることで、将来のブラウザーや支援技術でも解釈しやすい形にしておく**という考え方です。

日常生活でいえば、駅や公共施設の案内が、ある一社の特定端末でしか読めない独自記法ではなく、誰でも解釈しやすい記号や表記で設計されている状態に近いです。

Web でも同じで、独自実装を積み上げるほど、特定環境では動いても別の組み合わせで意味が崩れやすくなります。

堅牢は派手な改善ではありません。しかし、将来の保守コストと利用者の取りこぼしを減らすという意味で、かなり投資対効果の高い原則です。

# 開発で確認したい観点

堅牢のレビューでは、次の問いが有効です。

1. そのUI部品は、見た目どおりの role を持っているか
2. ラベルと accessible name は一致しているか
3. 現在値や選択状態、展開状態がプログラムで取得できるか
4. 非同期更新や保存結果、エラー通知が支援技術に伝わるか
5. 標準要素で足りるところを、不要にカスタム実装していないか

特にフロントエンド開発では、デザインシステムの部品単位でこの観点を持つと効果的です。ボタン、タブ、アコーディオン、コンボボックス、トースト、ダイアログなど、再利用される部品で意味情報が欠けていると、画面全体に同じ問題が広がるからです。

# まとめ

WCAG の「堅牢」は、壊れにくさの抽象論ではありません。

それは、**UI の意味と状態を、支援技術を含む多様なソフトウェアが解釈できるようにする設計原則**です。

改札機の残高表示が「今の自分の状態」なのか判然としないと不安になるように、Web の画面でも、見た目だけで成立している状態変化は一部のユーザーに届きません。

標準の HTML を使うこと、必要な name・role・value を正しく露出すること、ステータスメッセージを live region などで通知すること。この地味な積み重ねが、堅牢の実装です。

アクセシビリティの議論では、どうしても色やキーボード操作の話が目立ちます。しかし、実際の利用体験を最後まで支えるのは、「そのUIがソフトウェアにとっても意味を持っているか」という部分です。

堅牢は、ユーザーに見えない場所で体験の土台を支える原則だと言ってよいと思います。
