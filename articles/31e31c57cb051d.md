以前、私は次の記事で **GitHub Copilot CLI は「面で作る」作業に強く、VS Code の Copilot Chat Agent Mode は「点で整える」作業に強い**、という使い分けを整理しました。

https://zenn.dev/tomokusaba/articles/838cdac8d71e52

前回は、主に「どこで動くか」「どの粒度の変更に向くか」という観点で見ました。今回はその続きとして、もう少し細かい **カスタムエージェントの呼び出し方** に注目します。

前回記事では Instructions / Skills / Custom Agents / Prompt Files を横並びで比較しました。今回はその中でも Custom Agents に絞り、特に「どう呼び出すか」が作業の進め方にどう影響するかを見ていきます。

同じ「Reviewer に見てほしい」「Planner に計画させたい」という依頼でも、GitHub Copilot CLI では自然言語の文脈からカスタムエージェントを推測して呼び出せます。一方、VS Code Agent Mode では Agent ピッカーや `/` コマンドとして見えるカスタマイズを使い、ユーザーが明示的に役割を選びやすい設計になっています。

この記事は、前回記事を読んで Custom Agents の位置づけは把握したものの、実際に「自然言語で呼ぶべきか」「明示的に選ぶべきか」で迷っている方向けです。

この記事では、この違いを単なる好みではなく、**意図の明示性・コンテキスト共有・再現性・誤起動 / 呼び忘れ・実装精度** の観点で整理します🧭

:::message
本記事は 2026 年 4 月時点で確認できる公式ドキュメントをもとにした考察です。GitHub Copilot CLI / VS Code Agent Mode は更新が速いため、画面表示やコマンド名は利用環境によって変わる可能性があります。
:::

## 本記事のゴール

- GitHub Copilot CLI と VS Code Agent Mode におけるカスタムエージェントの呼び出し方の違いを整理する
- 自然言語呼び出しとスラッシュコマンド / 明示選択型の強み・弱みを理解する
- 呼び出し方の違いが、コード実装の精度にどう影響しそうかを考える
- Copilot CLI の `/fleet` を、並列化と利用量の両面から位置づける

## 前提: そもそも Custom Agents とは何か

GitHub Copilot CLI の公式ドキュメントでは、Custom Agents は「Copilot の特殊化されたバージョン」と説明されています。特定のワークフロー、コーディング規約、専門的なユースケースに合わせるための仕組みです。

CLI には Explore / Task / General-purpose / Code-review などの既定エージェントがあり、さらに Markdown の agent profile で独自エージェントを定義できます。公式ドキュメントでは、CLI でカスタムエージェントを使う方法として次の 3 つが挙げられています。

- `/agent` で利用可能なカスタムエージェント一覧から選ぶ
- `Use the refactoring agent ...` のように、自然言語で直接呼びかける
- `copilot --agent=refactor-agent --prompt "..."` のように、コマンドラインオプションで指定する

一方、VS Code の Custom Agents は `.agent.md` ファイルで定義します。役割（persona）、利用できるツール、モデル、handoff（エージェント間の引き継ぎ）などを宣言でき、チャットビューの Agent ピッカーから切り替えるのが基本です。VS Code 側では、Prompt Files や Skills なども `/` コマンドとして見えるため、チャット入力欄から**明示的に呼び出すカスタマイズ**を作りやすいのが特徴です。

:::message
VS Code の公式ドキュメントでは、Custom Agents 自体は主に Agent ピッカーで選ぶものとして説明されています。一方で、Prompt Files や Skills は `/` コマンドとして呼び出せます。本記事では、VS Code 側の「ユーザーが明示的な入口を選んで呼び出す」体験を、広めに **スラッシュコマンド / 明示選択型** と呼んで整理します。
:::

## 呼び出し型を 2 つに分けて考える

今回の主題を単純化すると、次の 2 つの呼び出し型の違いです。

| 観点 | 🖥️ GitHub Copilot CLI | 🧑‍💻 VS Code Agent Mode |
|------|----------------------|--------------------------|
| 呼び出しの主軸 | 自然言語から推測して呼び出す / `/agent` / `--agent` | Agent ピッカー、`/` コマンド、Prompt Files、Skills などで明示する |
| 典型例 | `Use the refactoring agent to ...` | `/tests`、自作 Prompt File / Skill としての `/review`、Agent ピッカーで Reviewer を選択 |
| 体験の重心 | 会話の流れを止めずに委譲する | 役割や定型タスクを明示してから実行する |
| 向いている場面 | 大きな依頼を自然文で投げ、Copilot に分解も任せたい | いまから何をするかを固定して、小さく確実に回したい |

この違いは、UI の差だけではありません。**どのタイミングで「誰に任せるか」を固定するのか** が違います。

CLI では、ユーザーの自然文を受け取ったメインエージェントが「これは refactoring agent に任せた方がよさそう」「ここは code-review agent が向いていそう」と判断できます。もちろん、`--agent` や `/agent` で明示することもできますが、自然言語の流れのまま委譲できるのが強みです。

Agent Mode では、エディター上の選択範囲、開いているファイル、Problems パネルなどの暗黙コンテキストを活かしながら、ユーザーが `/tests` や `/fix`、あるいは自作の Prompt File / Skill を選んでから依頼できます。**「このモードでやる」ことを先に固定しやすい**のが強みです。

## 観点 1: 意図の明示性

まず分かりやすいのは、意図の明示性です。

自然言語呼び出しは柔らかく、会話の流れを壊しません。

```text
Use the refactoring agent to make the service layer async-friendly.
After that, ask the reviewer agent to check the diff.
```

このように書くと、やりたいことと役割が同じ文の中に入ります。CLI の公式ドキュメントでも、特定のカスタムエージェントが向いていると分かっている場合は、プロンプトの中で直接呼び出せると説明されています。

一方、スラッシュコマンド / 明示選択型は、依頼の入口で意図が固定されます。

```text
/tests
選択中の UserService に対して、正常系・異常系・境界値のユニットテストを追加してください。
```

あるいは Reviewer エージェントを選んだ状態で、次のように投げます。

```text
この差分を、null 許容・例外処理・テスト不足の観点でレビューしてください。
```

前者は **「文脈で伝える」**、後者は **「入口で宣言する」** という違いです。

実装精度という観点では、明示性が高いほど「何をしてほしいか」はブレにくくなります。一方で、明示しすぎるとユーザーが毎回適切な入口を選ぶ必要があります。自然言語呼び出しはその負担が軽い代わりに、文が曖昧だとメインエージェントの解釈に依存します。

## 観点 2: コンテキスト共有

次に、コンテキスト共有です。

CLI はターミナル上でリポジトリ全体を相手にするため、前回の記事で書いたように **「面で作る」** 体験と相性が良いです。`copilot` を起動すると、信頼したディレクトリ配下のファイルを読み、必要に応じて編集やコマンド実行を進めます。大きな機能追加で、Controller / Service / Repository / Test を横断するような作業では、この広いスコープが効きます。

自然言語でカスタムエージェントを呼ぶ場合も、依頼文の中に「まず調査」「次に実装」「最後にレビュー」という流れを含めやすいです。メインエージェントが全体を握り、その中で適した担当者に任せるイメージです。

一方、VS Code Agent Mode はエディターの状態が強いコンテキストになります。公式ドキュメントでも、アクティブなファイル、現在の選択、ファイル名が暗黙のコンテキストとして含まれると説明されています。

このため、Agent Mode では次のような小さな依頼が強くなります。

- 選択中のメソッドだけを `/fix` する
- 開いている `.cs` ファイルに対して `/tests` を実行する
- Problems パネルのエラーを起点に修正する
- Reviewer エージェントに、現在の差分だけを見せる

つまり、CLI は **リポジトリ全体の文脈を渡しやすい**。Agent Mode は **いま見ているコードの文脈を渡しやすい**。この差が、実装の粒度に影響します。

## 観点 3: 再現性

再現性という点では、スラッシュコマンド / 明示選択型に分があります。

たとえばチームで次のような Prompt File や Skill を用意しておくとします。

```markdown
---
description: ASP.NET Core のサービス層に対するテストを生成する
---

あなたは C# / xUnit のテスト生成に詳しいエージェントです。
- 正常系、異常系、境界値を最低 1 つずつ含める
- 既存の命名規則に合わせる
- テストが不安定になる DateTime.Now などは避ける
```

これを `/service-tests` のように呼び出せる状態にしておけば、毎回プロンプトを長く書かなくても、同じ前提で依頼できます。実装精度は、モデルの性能だけでなく **同じ前提を毎回渡せるか** にも左右されます。

CLI でも `--agent=...` を使えば再現性は高められます。

```pwsh
copilot --agent=refactor-agent --prompt "Refactor the service layer to use async/await consistently."
```

ただし、自然言語で「refactoring agent を使って」と書く運用は、便利な反面、文の揺れが出ます。チームで同じ作業を繰り返すなら、CLI でも `--agent` や定型プロンプトをシェルスクリプト化する方が安定しやすいです。

## 観点 4: 誤起動と呼び忘れ

自然言語呼び出しのリスクは、**誤起動** と **呼び忘れ** です。

たとえば、次のような依頼をしたとします。

```text
Refactor the service layer and review the result.
```

人間としては「実装担当とレビュー担当を分けてほしい」というつもりでも、モデルがメインエージェントだけで処理してしまう可能性があります。GitHub のドキュメントでも、CLI のモデルはサブエージェントに委譲することもあれば、メインエージェントで直接処理することもある、と説明されています。

これは悪いことばかりではありません。小さい変更なら、わざわざ別エージェントを起動しない方が速く、コンテキストも単純です。ただし、**「必ず Reviewer を通したい」** という品質ゲートを置くなら、自然言語だけに任せるのは少し弱いです。

Agent Mode の明示選択型は、ここを補いやすいです。Reviewer エージェントを選ぶ、あるいは `/review` 相当の入口を用意しておけば、少なくとも「レビューとして実行する」意図は固定できます。

一方で、明示選択型には **呼び忘れ** があります。実装に集中していると、最後に Reviewer に切り替えるのを忘れる。`/tests` を打たずに「テストもお願い」と自然文で済ませてしまう。こうした運用ミスは十分に起こり得ます。

したがって、品質を安定させたい場合は、呼び出し型だけでなくワークフローとして組むのが大切です。

呼び忘れを防ぐワークフロー例としては、次のように Plan → Implement → Tests → Review を明示的に分ける形が考えられます。

```mermaid
flowchart LR
    Plan[🧭 Plan] --> Impl[🛠️ Implement]
    Impl --> Test[🧪 Tests]
    Test --> Review[🧐 Review]
    Review --> Fix[🔁 Fix if needed]
```

VS Code の Custom Agents には handoffs があるため、Planning → Implementation → Review のような順序を UI 上の導線にしやすいです。CLI では plan mode や `--agent`、自然言語での明示指示を組み合わせて、同じ流れをターミナル側に作るイメージになります。

## 総合整理: 呼び出し方は実装精度にどう影響するか

ここまでを踏まえると、実装精度への影響は次のように整理できます。

| 観点 | 自然言語呼び出しが効く場面 | スラッシュコマンド / 明示選択型が効く場面 |
|------|-----------------------------|------------------------------------------|
| 🧭 意図の明示性 | 大きな目的を会話のまま伝えたい | 役割やタスクを入口で固定したい |
| 🧠 コンテキスト共有 | リポジトリ全体の流れを見て判断したい | 選択範囲・開いているファイル・差分を正確に扱いたい |
| 🔁 再現性 | その場の判断で柔軟に委譲したい | チームの定型作業として同じ前提を毎回使いたい |
| ⚠️ 誤起動 / 呼び忘れ | 委譲判断をモデルに任せられるが、必ず分担されるとは限らない | 呼び出せば明確だが、ユーザーが選び忘れる可能性がある |
| 🎯 実装精度 | 大きなスコープの初動精度を上げやすい | 小さな変更・レビュー・テスト生成の安定度を上げやすい |

私の感覚では、**自然言語呼び出しは「探索と実装を一気に進める精度」** に効きます。要件がまだ少し曖昧で、どの役割が必要かも含めてエージェントに考えさせたいときに向いています。

一方、**スラッシュコマンド / 明示選択型は「狙った作業を外さない精度」** に効きます。レビュー、テスト生成、ドキュメント更新、アクセシビリティチェックのように、観点が決まっている作業ではこちらが安定しやすいです。

前回記事の言葉で言えば、CLI は **面で作る初動**、Agent Mode は **点で整える仕上げ** に強い。その理由の一部は、カスタムエージェントの呼び出し方にも表れているのだと思います。

## 補足: `/fleet` は「自然言語呼び出し」を横に広げる

Copilot CLI 側で触れておきたいのが、`/fleet` スラッシュコマンドです。

公式ドキュメントでは、`/fleet` は複雑なリクエストを小さな独立タスクに分解し、サブエージェントで並列実行するためのコマンドとして説明されています。メインの Copilot エージェントがオーケストレーターになり、依存関係を見ながら、並列化できる部分を同時に実行します。

```pwsh
copilot
> /fleet Implement the user search feature.
> Split the work into API changes, unit tests, and documentation updates.
> Use @reviewer for the final review pass.
```

`/fleet` が効きやすいのは、独立した調査・実装・レビューに分けられる作業です。

| 並列化しやすい作業 | `/fleet` で期待できること |
|--------------------|--------------------------|
| 🔎 複数領域の調査 | 各サブエージェントが別コンテキストで調べ、結果だけを統合する |
| 🧪 テスト追加 | Controller / Service / Repository など、対象を分けてテストを書ける |
| 🧱 複数ファイルのリファクタ | 依存が薄いファイル群を分担し、最後に統合できる |
| 🧐 レビュー | 実装担当とは別の視点で最終確認しやすい |

これは、自然言語呼び出しの強みを横方向に広げる機能だと捉えています。単に「速くなる」だけではなく、**独立したコンテキストを複数立てられる** ことが重要です。調査担当が読んだ大量の情報を、実装担当のコンテキストにすべて混ぜなくて済むからです。

ただし、注意点もあります。

:::message alert
`/fleet` はサブエージェントごとに LLM（大規模言語モデル）とのやり取りが発生するため、通常の 1 つのエージェントで処理するより、プレミアムリクエストの消費が増える場合があります。加えて usage limits（利用制限）の観点でも、並列ワークフローは token consumption（トークン消費量）を増やしやすいため、制限に近いときは控えることが推奨されています。
:::

GitHub の usage limits ドキュメントでも、制限に近づいているときは parallel workflows（並列ワークフロー）を減らすことが推奨されています。つまり `/fleet` は、**何でも並列化すればよい魔法ではなく、並列化の価値が利用量に見合うときに使う道具** です。

## 私ならこう使い分ける

前回記事の「面で作って、点で整える」に、今回の呼び出し型を重ねると、私の使い分けは次のようになります。

| したいこと | 選ぶもの | 理由 |
|------------|----------|------|
| 🏗️ 新機能の骨格を作る | Copilot CLI + 自然言語呼び出し | スコープが広く、どの担当に分けるかも含めて任せやすい |
| 🚀 大きな作業を分割して進める | Copilot CLI + `/fleet` | 独立した調査・実装・レビューを並列化できる |
| 🧭 計画を先に固める | CLI の plan mode / VS Code の Planner エージェント | 書き始める前に誤解を減らせる |
| 🧪 選択中のコードにテストを足す | Agent Mode + `/tests` や自作 Prompt File | 対象コンテキストが狭く、入口を固定しやすい |
| 🧐 差分をレビューする | Agent Mode + Reviewer エージェント | エディター上の差分確認と相性がよい |
| 🔁 チームの定型作業を繰り返す | Agent Mode の `/` コマンド / CLI の `--agent` | 再現性を高めやすい |

大事なのは、どちらか一方に寄せ切らないことです。

CLI の自然言語呼び出しは、**「大きな目的を渡して、必要な専門家を呼び分けてもらう」** 体験に向いています。Agent Mode の明示選択型は、**「いまからこの観点で処理する」** ことを固定し、狙った差分を出す体験に向いています。

コード実装の精度は、モデルだけで決まるわけではありません。**いつ、どの入口で、どの文脈を渡すか** にも大きく左右されます。カスタムエージェントの呼び出し方は、その精度を調整するレバーの 1 つだと感じています。

## おわりに

前回は、GitHub Copilot CLI と VS Code Agent Mode を「作業粒度」で比較しました。今回はそこから一歩進んで、**カスタムエージェントをどう呼び出すか** という観点で見直しました。

- Copilot CLI は、自然言語の流れの中でカスタムエージェントを呼び出せる
- Agent Mode は、Agent ピッカーや `/` コマンドとして見えるカスタマイズを使い、入口を明示しやすい
- 自然言語呼び出しは、広いスコープの探索・実装に向く
- スラッシュコマンド / 明示選択型は、レビュー・テスト・小さな修正の再現性に向く
- `/fleet` は独立した調査・実装・レビューを並列化できる一方、利用量 / プレミアムリクエストが増えやすい点に注意する

「どちらが優れているか」ではなく、**どの呼び出し方なら、今回の作業の意図がいちばん正確に伝わるか** で選ぶのがよさそうです。

大きく作るときは CLI に自然言語で任せます。独立タスクが多いなら `/fleet` で横に広げます。最後は VS Code Agent Mode で、Reviewer エージェントや `/tests` を使って点で整えます。

この組み合わせが、現時点の私にとっていちばん「レビューに出しやすいコード」に近づくための流れです🛠️

それでは、よい Copilot ライフを！
