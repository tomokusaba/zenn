## はじめに

REST API と gRPC の違いを説明する記事を読んでいて、少し補足したくなったポイントがありました。

https://zenn.dev/rin__rin/articles/dba550bfa3c78f

ざっくり言うと、「REST API は JSON などのテキスト、gRPC は Protocol Buffers のバイナリ」という整理です。実務上の説明としては分かりやすいのですが、厳密には **REST API / HTTP はテキストしか送れないわけではありません**。

この記事は元記事の主旨を否定するものではなく、HTTP の性質について補足する記事です。
もしかすると、REST API そのものというより、OpenAPI でよく見る JSON 中心の API 定義や、Swagger UI で試しやすい形式と混ざっているのかもしれません。

HTTP のメッセージボディはバイト列を運べます。JSON を送ることが多いだけで、画像、PDF、MessagePack、Protocol Buffers、そして今回使う MemoryPack のようなバイナリも送れます📦

この記事では、ASP.NET Core Minimal API で MemoryPack のバイナリを受け取って返す、REST API 風のエンドポイントを作ります。さらに、その API を Blazor WebAssembly から呼び出す最小限のコードも載せます。

## 本記事のゴール

- REST API が JSON / テキスト専用ではないことを確認する
- Minimal API で `application/x-memorypack` のバイナリを受け取って返す
- Blazor WebAssembly の `HttpClient` から `ByteArrayContent` でバイナリを POST する
- gRPC と REST の違いを「バイナリかどうか」だけで語らない視点を持つ

## 前提条件

- ✅ .NET SDK 10.0.300（2026-05-31 時点）
- ✅ ASP.NET Core Minimal API
- ✅ Blazor WebAssembly
- ✅ MemoryPack 1.21.4
- ✅ Windows + PowerShell で検証

```text
RestBinaryProof

```

## REST API は「テキスト専用」ではない

まず前提を整理します。

REST はアーキテクチャスタイルであり、「ボディは JSON でなければならない」という制約ではありません。Web API の文脈では JSON が圧倒的に多いので、REST API = JSON という印象になりがちですが、HTTP として見ればボディはバイト列です。

RFC 9110 では、HTTP メッセージの content について次のように説明されています。

> HTTP messages often transfer a complete or partial representation as the message "content": a stream of octets sent after the header section, as delineated by the message framing.
>
> — [RFC 9110: HTTP Semantics, 6.4 Content](https://www.rfc-editor.org/rfc/rfc9110.html#name-content)

ここで重要なのは `a stream of octets`、つまりオクテット列（バイト列）です。あとは `Content-Type` で、そのバイト列をどう解釈するかを伝えます。

| 通信 | Content-Type の例 | ボディ |
|------|-------------------|--------|
| 📝 JSON API | `application/json` | UTF-8 JSON |
| 🖼️ 画像アップロード | `image/png` | PNG バイナリ |
| 📄 PDF ダウンロード | `application/pdf` | PDF バイナリ |
| 📦 今回の例 | `application/x-memorypack` | MemoryPack バイナリ |

つまり、gRPC がバイナリを使うことは事実ですが、バイナリを送れること自体は gRPC 専用の特徴ではありません。

## MemoryPack を使う理由

今回はバイナリシリアライザーとして [MemoryPack](https://github.com/Cysharp/MemoryPack) を使います。

MemoryPack は Cysharp 製の C# / Unity 向けバイナリシリアライザーです。README では "Zero encoding extreme performance binary serializer for C# and Unity." と説明されており、Source Generator ベースでシリアライズコードを生成します。

今回の主題はベンチマークではなく、REST API でもバイナリペイロードを普通に扱えることの実証です。そのため、コード量が少なく、C# 同士で DTO を共有しやすい MemoryPack を選びました。

:::message alert
MemoryPack は C# に強く寄せたバイナリ形式です。外部公開 API や多言語クライアントを前提にする場合は、スキーマ互換性、バージョニング、ツールチェーン、デバッグ容易性を含めて選定してください。単に「JSON より速そう」だけで置き換えるものではありません。
:::

## Step 1: プロジェクトを作成する

まずは検証用のソリューションを作ります。

```pwsh
mkdir RestBinaryProof
cd RestBinaryProof

dotnet new sln -n RestBinaryProof

dotnet new classlib -n Shared -f net10.0
dotnet new webapi -n Api -f net10.0 --no-https --no-openapi --use-minimal-apis
dotnet new blazorwasm -n Client -f net10.0

dotnet sln add .\Shared\Shared.csproj
dotnet sln add .\Api\Api.csproj
dotnet sln add .\Client\Client.csproj

dotnet add .\Api\Api.csproj reference .\Shared\Shared.csproj
dotnet add .\Client\Client.csproj reference .\Shared\Shared.csproj

dotnet add .\Shared\Shared.csproj package MemoryPack --version 1.21.4
dotnet add .\Api\Api.csproj package MemoryPack --version 1.21.4
dotnet add .\Client\Client.csproj package MemoryPack --version 1.21.4
```

`.NET 10` の SDK では、環境によってソリューションファイルが `.sln` ではなく `.slnx` になる場合があります。`dotnet sln add` や `dotnet build` はそのまま使えます。

## Step 2: Shared に MemoryPack 対応 DTO を置く

`Shared\Contracts.cs` を作成します。

```csharp
using MemoryPack;

namespace Shared;

[MemoryPackable]
public partial record CreateTodoRequest(string Title, int Priority);

[MemoryPackable]
public partial record CreateTodoResponse(Guid Id, string Title, int Priority, DateTimeOffset CreatedAt);
```

ポイントは `[MemoryPackable]` と `partial` です。MemoryPack は Source Generator でシリアライズ用コードを生成するため、対象型にこれらの指定が必要です。

## Step 3: Minimal API でバイナリを受け取って返す

次に `Api\Program.cs` を次のようにします。

```csharp
using MemoryPack;
using Shared;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddCors(options =>
{
    options.AddDefaultPolicy(policy =>
        policy.AllowAnyOrigin()
            .AllowAnyHeader()
            .AllowAnyMethod());
});

var app = builder.Build();

app.UseCors();

app.MapPost("/binary/todos", async (HttpRequest httpRequest, CancellationToken cancellationToken) =>
{
    var request = await MemoryPackSerializer.DeserializeAsync<CreateTodoRequest>(
        httpRequest.Body,
        cancellationToken: cancellationToken);

    if (request is null)
    {
        return Results.BadRequest("Request body is empty or invalid.");
    }

    var response = new CreateTodoResponse(
        Guid.NewGuid(),
        request.Title,
        request.Priority,
        DateTimeOffset.UtcNow);

    var bytes = MemoryPackSerializer.Serialize(response);
    return Results.Bytes(bytes, "application/x-memorypack");
});

app.Run();
```

ここでは、HTTP リクエストの `Body` をそのまま MemoryPack でデシリアライズしています。レスポンスも `MemoryPackSerializer.Serialize` で `byte[]` にして、`Results.Bytes` で返しています。

`Results.Bytes` は ASP.NET Core の API で、公式ドキュメントでも "Writes the byte-array content to the response." と説明されています。

[Results.Bytes Method](https://learn.microsoft.com/ja-jp/dotnet/api/microsoft.aspnetcore.http.results.bytes?view=aspnetcore-10.0&WT.mc_id=DT-MVP-5004827)

:::message alert
`AllowAnyOrigin()` はローカル検証用です。実運用では許可する Origin、メソッド、ヘッダーを絞ってください。
:::

## Step 4: Blazor WebAssembly から ByteArrayContent で送る

Blazor WebAssembly 側では、まず `Client\Program.cs` の `HttpClient` の接続先を Minimal API にします。

```csharp
using Microsoft.AspNetCore.Components.Web;
using Microsoft.AspNetCore.Components.WebAssembly.Hosting;
using Client;

var builder = WebAssemblyHostBuilder.CreateDefault(args);
builder.RootComponents.Add<App>("#app");
builder.RootComponents.Add<HeadOutlet>("head::after");

builder.Services.AddScoped(_ => new HttpClient { BaseAddress = new Uri("http://localhost:5000") });

await builder.Build().RunAsync();
```

次に `Client\Pages\Home.razor` を最小限の呼び出しコードにします。

```razor
@page "/"
@using System.Net.Http.Headers
@using MemoryPack
@using Shared
@inject HttpClient Http

<PageTitle>Home</PageTitle>

<h1>MemoryPack over REST</h1>

<p>Minimal API に MemoryPack のバイナリを POST します。</p>

<button @onclick="SendAsync" disabled="@isSending">送信</button>

@if (createdTodo is not null)
{
    <dl>
        <dt>ID</dt>
        <dd>@createdTodo.Id</dd>
        <dt>Title</dt>
        <dd>@createdTodo.Title</dd>
        <dt>Priority</dt>
        <dd>@createdTodo.Priority</dd>
        <dt>CreatedAt</dt>
        <dd>@createdTodo.CreatedAt</dd>
    </dl>
}

@code {
    private bool isSending;
    private CreateTodoResponse? createdTodo;

    private async Task SendAsync()
    {
        isSending = true;

        try
        {
            var request = new CreateTodoRequest("REST can send binary", 1);
            var payload = MemoryPackSerializer.Serialize(request);

            using var content = new ByteArrayContent(payload);
            content.Headers.ContentType = new MediaTypeHeaderValue("application/x-memorypack");

            using var response = await Http.PostAsync("binary/todos", content);
            response.EnsureSuccessStatusCode();

            var responseBytes = await response.Content.ReadAsByteArrayAsync();
            createdTodo = MemoryPackSerializer.Deserialize<CreateTodoResponse>(responseBytes);
        }
        finally
        {
            isSending = false;
        }
    }
}
```

`ByteArrayContent` は、名前のとおりバイト配列を HTTP content として扱うためのクラスです。公式ドキュメントでも "Provides HTTP content based on a byte array." と説明されています。

[ByteArrayContent Class](https://learn.microsoft.com/ja-jp/dotnet/api/system.net.http.bytearraycontent?view=net-10.0&WT.mc_id=DT-MVP-5004827)

Blazor から Web API を呼ぶ基本形については、Microsoft Learn の Blazor ドキュメントも参考になります。

[Call a web API from ASP.NET Core Blazor](https://learn.microsoft.com/ja-jp/aspnet/core/blazor/call-web-api?view=aspnetcore-10.0&WT.mc_id=DT-MVP-5004827)

## REST と gRPC の違いはどこにあるのか

ここまで見ると、「では gRPC と REST は同じなの？」と思うかもしれません。もちろん同じではありません。

ただし、違いを「テキスト vs バイナリ」だけに置くと、少し雑になります。

| 観点 | 🧭 REST API over HTTP | ⚡ gRPC |
|------|----------------------|--------|
| ペイロード | JSON もバイナリも送れる | 通常は Protocol Buffers |
| インターフェイス | リソースと HTTP メソッドで表現することが多い | service / rpc と `.proto` で定義 |
| スキーマ | OpenAPI などを使えるが必須ではない | `.proto` が中核 |
| ブラウザー | `fetch` / `HttpClient` で扱いやすい | ネイティブ gRPC はブラウザーから直接扱いにくい |
| ストリーミング | HTTP と実装次第 | 一方向 / 双方向ストリーミングがモデル化されている |

gRPC の強みは、Protocol Buffers によるスキーマ、コード生成、サービス定義、HTTP/2 を前提にしたストリーミングなどが一体になっている点です。

一方で、REST API でも `Content-Type` を適切に決めれば、バイナリペイロードは普通に送れます。今回の `application/x-memorypack` はその一例です。

## 注意点

バイナリ REST API は可能ですが、どの場面でも JSON の代わりに使えばよいわけではありません。

| 項目 | 注意点 |
|------|--------|
| 🔍 デバッグ | DevTools やログで中身を直接読みづらい |
| 🧩 互換性 | メンバーの順序やバージョン管理を意識する必要がある |
| 🌐 クライアント | C# 以外のクライアントがあるなら形式選定が重要 |
| 📚 API 仕様 | OpenAPI で表現しづらい場合がある |
| 🛡️ セキュリティ | 入力サイズ制限、認証、検証は JSON と同様に必要 |

特に MemoryPack はメンバーの順序が重要な形式です。外部公開 API にするなら、DTO の変更ルールや後方互換性をチームで決めておく必要があります。

## まとめ

REST API は JSON を送ることが多いですが、テキストしか送れないわけではありません。HTTP の content はバイト列であり、`Content-Type` によってその解釈を決めます。

Minimal API では `HttpRequest.Body` から MemoryPack を直接デシリアライズでき、レスポンスも `Results.Bytes` でバイナリとして返せます。Blazor WebAssembly からも `ByteArrayContent` を使えば、同じようにバイナリを POST できます📦

gRPC と REST の比較では、「gRPC はバイナリ、REST はテキスト」と覚えるよりも、「gRPC はスキーマ・コード生成・通信モデルを含む RPC フレームワーク、REST は HTTP のリソース指向の使い方」と捉える方が、設計判断を間違えにくいと感じました。
