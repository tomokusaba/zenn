# はじめに

Webアクセシビリティというと、多くの開発者はまず色のコントラスト、キーボード操作、スクリーンリーダー対応を思い浮かべると思います。

もちろんそれらは重要です。しかし、実際の利用体験を壊しやすいのは、もっと手前にある「この文面は理解できるか」「次に何が起こるか予測できるか」「間違えたときに立て直せるか」という問題です。

WCAG 2.2 では、これを原則 3 の **Understandable（理解可能）** として整理しています。W3C はこの原則を「情報およびユーザーインターフェイスの操作は理解可能でなければならない」と説明しています。

この原則は、実は日常生活の体験に置き換えるととても理解しやすくなります。たとえば、駅で見慣れない略語だらけの案内板に出会ったとき、病院の説明で専門用語が連発されたとき、オンラインの申込フォームで「入力が正しくありません」とだけ表示されたとき、私たちは単に「見えない」のではなく「わからない」という壁にぶつかります。

この記事では、WCAG における「理解可能」の原則を日常生活の例から読み解きながら、特に 3.1.3 の「一般的でない用語」と 3.1.4 の「略語」に注目します。そのうえで、Microsoft Learn が実際にどのように用語集、用語標準化、初出定義のガイドラインを整備しているのかを紹介します。

# WCAG の「理解可能」とは何か

WCAG 2.2 の原則 3 は、次の 3 つの観点で構成されています。

1. 読みやすく理解しやすいこと
2. 挙動が予測可能であること
3. 入力時の失敗を防ぎ、修正しやすいこと

それぞれ WCAG では次のガイドライン群に対応しています。

- 3.1 Readable
- 3.2 Predictable
- 3.3 Input Assistance

この 3 つを、私はそれぞれ次のように考えると腹落ちしやすいと思っています。

- Readable は「言葉が通じるか」
- Predictable は「振る舞いが読めるか」
- Input Assistance は「失敗しても戻ってこられるか」

アクセシビリティの議論では、どうしても「利用できるかどうか」が中心になりがちです。しかし現実の UX では、「利用できる」だけでは不十分です。読めること、意味が取れること、迷わないこと、やり直せることまで含めて初めて、その体験は理解可能になります。

# 駅の案内板で考える Readable

たとえば大阪の人であれば、大阪環状線のホームで電車の行き先表示を見た瞬間に、「この電車で目的の駅まで行けるか」をかなり自然に判断できるでしょう。

しかし、他の地域から来た人にとってはそう簡単ではありません。たとえば「関西空港行き」や「紀州路快速」と表示されていても、どの駅までは環状線を回り、どこから先で環状線を外れていくのかを即座に判断できないことがあります。

地元の人にとっては常識でも、初めて使う人にとっては、表示されている言葉だけでは行動を決めるのに十分ではないわけです。

私は、大阪環状線の行き先表示を見ていつも困惑しています。手元の乗換案内アプリで本当にたどり着けるのかどうか毎回検索しています。

Web でも同じです。文章が読めることと、意味が取れることは別問題です。W3C は Guideline 3.1 Readable の意図として、「テキストをユーザーや支援技術が読めること、そして理解に必要な情報が利用可能であること」を挙げています。

## 3.1.3 一般的でない用語

SC 3.1.3 Unusual Words では、慣用句、専門用語、限定的な意味で使われる語について、**その定義を特定できるメカニズムが利用可能であること**を求めています。

W3C の Understanding では、これが必要になる例として次のようなものが挙げられています。

- idioms（慣用句）
- jargon（専門用語）
- unusual or restricted usage（一般的でない、または限定的な使い方）

つまり、単に難しい単語を使うなという話ではありません。使うのであれば、利用者がその意味に到達できる導線を用意しなければならない、という考え方です。

日常生活でいえば、博物館で専門用語の横に短い注釈が付いている展示や、登山道の看板に地域固有の地形名の説明が添えられている状態に近いです。言葉そのものを禁止するのではなく、理解の足場を用意するわけです。

W3C は実装上の手法として、次のような例を示しています。

- 用語定義へのリンク
- インライン定義
- 用語集の提供
- オンライン辞書検索への導線

## 3.1.4 略語

SC 3.1.4 Abbreviations では、**略語の正式名称または意味を特定できるメカニズム**を求めています。

病院の受付で「CT は 2 階、MRI は午後、再診は内科外来へ」とだけ言われたとき、医療者でなければ一瞬止まります。企業の業務システムでも、「SSO」「RBAC」「ETL」「SKU」のような略語が当然の前提として並ぶことがあります。

W3C は略語が問題になる理由として、次のような事情を挙げています。

- 略語は通常の単語の読み方で発音できないことがある
- 同じ略語が文脈によって別の意味を持つことがある
- 認知障害や学習障害のある人にとって意味推定の負荷が高い
- 拡大表示では文脈が見えにくくなり、略語の意味を補いにくい

これも日常ではよくある体験です。社内チャットで「FYI」「ETA」「ASAP」が飛び交っているとき、新しく入ったメンバーだけが会話速度についていけないことがあります。用語を知っている人にとっては効率化ですが、知らない人にとっては情報の入口そのものが閉じます。

## HTML ではどう表現できるか

理解可能のためのメカニズムは、派手な仕組みである必要はありません。多くは小さな設計判断で実現できます。

```html
<p>
	このシステムは
	<abbr title="Single Sign-On">SSO</abbr>
	に対応しています。
</p>

<p>
	<dfn id="rbac">RBAC</dfn>
	は、役割に基づいて権限を管理する方式です。
</p>

<p>
	詳しくは <a href="#glossary">用語集</a> を参照してください。
</p>

<dl id="glossary">
	<dt>RBAC</dt>
	<dd>Role-Based Access Control。役割ごとにアクセス権を制御する考え方。</dd>
</dl>
```

大事なのは、実装テクニックそのものよりも、「この語は初見の人に伝わるか」という視点を持つことです。

# Microsoft Learn は用語の壁をどう扱っているか

ここで面白いのが、Microsoft Learn がこの問題に対してかなり実務的に取り組んでいる点です。しかもそれは「アクセシビリティ特集ページ」だけでなく、日々のドキュメント運用そのものに埋め込まれています。

## 1. 略語は初出で定義する

Microsoft Learn の Writing style では、Abbreviations に関して次の方針が明示されています。

- ユーザーが略語を知っていると決めつけない
- 新しい略語は最初に出る場所で必ず定義する
- 似た略語を乱用しない
- 英語を第二言語として使うユーザーやローカライズを考えるなら略語を避ける

これはそのまま WCAG 3.1.4 の発想と重なります。つまり Microsoft Learn では、略語を使うこと自体を否定しているのではなく、**利用者が意味に到達できるように最初の接点を丁寧に設計する**ことを重視しているわけです。

開発者の感覚では、略語は情報を圧縮してくれる便利な道具です。しかし、初見の利用者にとっては、略語は「意味の欠損」でもあります。Microsoft Learn のガイドラインは、その欠損を最初の一回で埋めるという運用ルールだと言えます。

## 2. 用語検索と用語集を実際に提供している

Microsoft は [Microsoft Terminology](https://learn.microsoft.com/globalization/reference/microsoft-terminology) を公開しており、製品関連の承認済み用語を検索できる Microsoft Terminology Search も案内しています。

また、個別の製品領域では実際に glossary ページが提供されています。たとえば [Microsoft 365 Glossary](https://learn.microsoft.com/microsoft-365/community/glossary) では、Microsoft 365 周辺で頻出する用語や略語を一覧化し、「terms and acronyms を日々使っているが、それを理解すること自体が半分の戦いだ」と明言したうえで、意味の曖昧さを解消しようとしています。

これは WCAG 3.1.3 と 3.1.4 で W3C が挙げている「用語集」「辞書検索」「意味へのリンク」というメカニズムに非常に近い実践です。

つまり Microsoft Learn の取り組みは、単に文章をわかりやすく書くという話に留まりません。**専門用語が存在することを前提に、その理解コストを下げるための補助線を公開された仕組みとして用意している**点に価値があります。

## 3. 「平易な表現」を UX として扱っている

Writing style では、Microsoft は simple and straightforward language、つまり平易で率直な表現をよいユーザー体験の基礎として扱い、「ユーザーが理解できない用語を使わないこと」や「不要に特殊な語を持ち込まないこと」を勧めています。

ここが重要です。難しい言葉を避けることは、単なる文体の好みではありません。UX の一部であり、アクセシビリティの一部でもあります。

大阪環状線の案内でも、表示文字が大きいだけでは足りません。「関西空港行き」や「紀州路快速」という語が、その場の利用者にとってどんな経路を意味するのか分からなければ、次に動けません。同じように Web でも、ARIA 属性が正しくても、説明文が専門用語だらけなら理解可能とは言えません。

# コンビニのレジで考える Predictable

次に 3.2 Predictable です。

セルフレジを使っていて、商品をスキャンするたびにボタンの位置や画面遷移の規則が変わったらどうでしょうか。ある画面では右下が「次へ」なのに、次の画面では右下が「キャンセル」になっていたら、多くの人が迷います。

Web でも同じです。理解可能とは、文言だけの問題ではありません。**同じ操作には同じ結果が返る**という予測可能性も必要です。

WCAG 3.2 には次のような基準があります。

- 3.2.1 On Focus: フォーカスしただけで文脈を勝手に変えない
- 3.2.2 On Input: 入力しただけで意図しない画面遷移を起こさない
- 3.2.3 Consistent Navigation: 繰り返し登場するナビゲーションは同じ相対順序で出す
- 3.2.4 Consistent Identification: 同じ機能を持つものは一貫した識別で示す

日常生活では、駅の券売機や ATM の UI が近い感覚です。毎回ボタン配置が変わる券売機は、見えていても理解しにくいのです。

Web アプリケーションでは、次のような場面でこの問題が起きがちです。

- 検索条件の select を変えた瞬間に自動送信される
- 各ページで「保存」「更新」「適用」の使い分けが曖昧
- グローバルナビゲーションの順序がページごとに入れ替わる
- 同じアイコンがページによって別の意味を持つ

こうした不一致は、学習コストを上げ、利用者に「この画面は毎回読み直さなければ危ない」という緊張を強います。理解可能とは、利用者の記憶や注意力を無駄に消耗させない設計でもあります。

# 役所の申請フォームで考える Input Assistance

最後に 3.3 Input Assistance です。

役所や学校の申請フォームで、送信後に「エラーがあります」とだけ表示され、どこが悪いのか分からない経験はないでしょうか。入力欄が 20 個ある中で、1 箇所だけミスがあっても、それが見つからなければやり直しに近い負担になります。

WCAG の 3.3 系は、こうした「間違いの説明責任」を UI 側に求めています。

- 3.3.1 Error Identification: どこがエラーかを示す
- 3.3.2 Labels or Instructions: 必要な入力にはラベルや説明を付ける
- 3.3.3 Error Suggestion: 修正方法が分かるなら提案する
- 3.3.4 Error Prevention: 法的、金銭的、重要データでは誤送信を防ぐ
- 3.3.7 Redundant Entry: 同じ情報の再入力を強制しない
- 3.3.8 Accessible Authentication: 認知負荷の高い認証だけに頼らない

これも、見えるかどうかの問題ではなく、理解して立て直せるかどうかの問題です。

たとえば良いエラーメッセージは次のようなものです。

```text
郵便番号は 7 桁の半角数字で入力してください。
例: 1000001
```

悪いメッセージは次のようなものです。

```text
入力が正しくありません。
```

後者は、情報としてほとんど何も渡していません。利用者はエラーの存在は知れても、修正方法には到達できません。

Microsoft Learn の Writing style が、エラーメッセージについて「専門用語を避ける」「何が起きたかを伝える」「現実的な解決策を示す」としているのも、この Input Assistance の考え方と強く共鳴しています。

# 実装者のための設計ポイント

「理解可能」は抽象的に見えますが、設計レビューではかなり具体化できます。私は最低限、次の観点で確認するのが有効だと思っています。

## 文言設計

- 専門用語、慣用句、社内略語を使っていないか
- 使う場合は初出定義、注釈、用語集への導線があるか
- エラーメッセージが原因と対処を説明しているか

## 画面挙動

- 入力中に文脈が勝手に変わらないか
- 同じ操作要素がページ間で一貫しているか
- ナビゲーションの順序や名称が安定しているか

## 入力支援

- 必須項目、形式、桁数、例示が事前に示されているか
- エラー箇所が明確で、修正方法まで案内できているか
- 重要操作で確認、取消、修正の機会があるか

アクセシビリティレビューではしばしば DOM、ARIA、コントラストなどが中心になりますが、実際にはこの「言葉の設計」と「エラー時の支援」が体験品質を大きく左右します。

# おわりに

WCAG の「理解可能」は、見えない人だけのための原則ではありません。知らない駅に降りた旅行者、専門外の説明を受ける患者、慣れないオンライン申請をする市民、新しいプロダクトに触れるユーザー全員に関わる原則です。

3.1.3 の「一般的でない用語」や 3.1.4 の「略語」は、一見すると細かな項目に見えます。しかし実際には、ユーザーが情報世界に入れるかどうかを決める入口です。意味に到達できない言葉は、リンクが切れているのとほとんど同じです。

その意味で、Microsoft Learn の取り組みは示唆的です。略語は初出で定義する。専門用語は用語集や用語検索に接続する。jargon を減らし、平易に書く。これは単なる文書品質の改善ではなく、理解可能性を組織的に運用する実践です。

Web アクセシビリティを考えるとき、私たちはつい「読めるか」「操作できるか」を先に見ます。しかしその次の問いとして、ぜひ「理解できるか」を置いてみてください。そこから見えてくる改善は、思っている以上に多いはずです。
