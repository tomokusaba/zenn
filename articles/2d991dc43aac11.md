# はじめに

2026年2月28日に開催される [.NETラボ勉強会 2026年2月](https://dotnetlab.connpass.com/event/378424/) において、筆者は「**Webアクセシビリティ技術と実装の実際**」というセッションで登壇します🎤

このセッションでは、実際に視覚障碍者の方にご登壇いただき、PCやスマートフォンをどのように操作しているのかをデモンストレーションしていただく構成を準備しています。Webアクセシビリティについて理論やガイドラインを解説するセッションは数多く存在しますが、**当事者の実際の操作を開発者が直接目にする機会**は極めて稀であり、だからこそ今回のセッションでその場を設けたいと考えました👀

本記事では、セッションに先立ち、なぜ「当事者によるデモンストレーション」がWeb開発者にとって技術的に重要な意味を持つのかを、WCAG 2.2の仕様、支援技術の仕組み、そしてこれまでの筆者自身の経験を踏まえて考察します📝

# 自動テストの限界 ― 機械が検出できない30〜40%の壁

Webアクセシビリティの品質を担保する手段として、axe-coreやLighthouseなどの自動テストツールは欠かせない存在です。筆者も以前の記事で、Playwright C#とaxe-coreを組み合わせたCI/CDパイプラインの構築方法を解説しました🔧

しかしここで直視すべき事実があります。自動テストツールが検出できるアクセシビリティ上の問題は、**全体の約30〜40%に過ぎない**という点です。W3Cの[Accessibility Evaluation Resources](https://www.w3.org/WAI/eval/Overview.html)においても、自動テストだけでは不十分であり、手動テストとの組み合わせが不可欠であることが明記されています🧪

たとえば、`alt` 属性が存在するかどうかは自動テストで検出できます。しかし、その `alt` テキストが画像の内容を正確に伝えているかは、機械では判定できません。フォーカスインジケーターがCSSで定義されているかは検出できても、そのインジケーターが実際の操作において十分に視認可能かは、人間の目で見なければわからないのです👓

この「残りの60〜70%」を検証するために最も有効な手段こそ、**実際の支援技術を使って操作する当事者の声を聞くこと**にほかなりません🗣️

# スクリーンリーダーの動作原理 ― 開発者が理解すべき「もう一つのレンダリング」

視覚障碍者がPCやスマートフォンを操作する際の中核的な支援技術がスクリーンリーダーです。代表的なスクリーンリーダーとしては、Windows環境では[Microsoft Narrator](https://support.microsoft.com/help/22798)や[NVDA（NonVisual Desktop Access）](https://www.nvaccess.org/)、[JAWS（Job Access With Speech）](https://www.freedomscientific.com/Products/Blindness/JawsDocumentation)が、macOS/iOS環境では[VoiceOver](https://www.apple.com/accessibility/mac/vision)が、Android環境では[TalkBack](https://support.google.com/accessibility/android/answer/6283677)がそれぞれ広く利用されています🔊

## アクセシビリティツリーという「見えないDOM」

ブラウザがHTMLをレンダリングする際、画面に表示されるビジュアルなレンダリングツリーとは別に、**アクセシビリティツリー**と呼ばれるもう一つのツリー構造を生成しています。Microsoftの[UI Automation API](https://learn.microsoft.com/windows/win32/winauto/entry-uiauto-win32)のドキュメントに詳述されているとおり、このアクセシビリティツリーは各UI要素の名前（Accessible Name）、役割（Role）、状態（State）、値（Value）といった情報をプログラム的に公開し、スクリーンリーダーはこのツリーを走査することでユーザーに情報を伝達します🌳

つまり、視覚的に美しく整ったWebページであっても、アクセシビリティツリー上の情報が不適切であれば、スクリーンリーダーのユーザーにとってはまったく異なる——しばしば理解不能な——体験になり得るということです。これは**ブラウザがHTMLを二重にレンダリングしている**という構造的な事実を理解しなければ、開発者は見落としてしまうポイントです⚠️

## 読み上げの順序問題

スクリーンリーダーは画面上の視覚的な配置ではなく、DOMの順序に従ってコンテンツを読み上げます。CSSのFlexboxで `order` プロパティを使って視覚的な表示順を変更していても、スクリーンリーダーはDOMの記述順に情報を提示します📄

この差異は、開発者がブラウザの画面だけを見ている限り決して発見できません。実際にスクリーンリーダーで読み上げを聞いて初めて、「視覚的には論理的な配置だが、読み上げ順は支離滅裂になっている」という状態に気づくことができるのです🎧

# 「見る」と「聞く」の認知モデルの違い

デモンストレーションを通じて開発者が最も強く受ける衝撃は、おそらく**情報取得の認知モデルそのものの違い**にあるでしょう🧠

## 視覚は並列、聴覚は直列

視覚によるWebページの閲覧は本質的に**並列的**です。ユーザーはページ全体を一瞬で俯瞰し、視線を自由に動かして必要な情報に直接アクセスすることができます。ヘッダー、サイドバー、メインコンテンツ、フッターといった構造的な区分けも、視覚的なレイアウトから瞬時に把握できます👁️

一方、スクリーンリーダーによる情報取得は本質的に**直列的**です。テキストは先頭から順に読み上げられ、ユーザーは一度に一つの情報しか受け取ることができません。ページ全体を「俯瞰」するためには、見出しジャンプ（`h1`〜`h6` 要素間の移動）やランドマークナビゲーション（`<nav>`、`<main>`、`<aside>` などのセマンティック要素間の移動）といった操作を駆使して、構造的な骨格を把握する必要があります🔄

この根本的な認知モデルの差異が、Webサイトの設計に及ぼす影響は計り知れません。たとえば、ページの冒頭に30個のナビゲーションリンクが並ぶサイトを想像してみてください。視覚的にはサイドバーにコンパクトに収まっていても、スクリーンリーダーのユーザーはメインコンテンツに到達する前にそれらすべてのリンクを聞かされることになります。WCAG 2.2の達成基準2.4.1（ブロックスキップ）が「スキップナビゲーション」リンクの提供を求めているのは、まさにこの問題に対処するためです⏭️

## 見出し構造の決定的な重要性

視覚的なWebページでは、フォントサイズや太さの違いで見出しの階層構造を直感的に理解できます。しかしスクリーンリーダーのユーザーにとって、見出し要素（`<h1>`〜`<h6>`）は**ページの目次そのもの**として機能します📚

NVDAやJAWSでは、`H` キーを押すことで次の見出しへジャンプし、`1`〜`6` の数字キーで特定のレベルの見出しに直接移動することができます。つまり、見出しの階層が適切に構造化されていれば、スクリーンリーダーのユーザーはページ内を効率的にナビゲートできます。逆に、見出し要素を使わずに `<div>` と CSSだけで見出しの見た目を実現しているページでは、スクリーンリーダーのユーザーはページの構造を把握する手がかりを完全に失います🚫

これは仕様書で読む知識としては理解できますが、**実際にスクリーンリーダーを使っている方がページ内を見出しジャンプで瞬時に移動する様子**を目の当たりにすると、見出し構造の設計がいかにクリティカルであるかが体感として腹落ちするのです💡

# スマートフォンにおける支援技術 ― タッチスクリーンという逆説

スマートフォンにおける視覚障碍者の操作は、PCの場合以上に開発者の想像を超えた世界です📱

iOSのVoiceOverやAndroidのTalkBackが有効な場合、タッチスクリーンの操作体系は根本的に変化します。通常のタップは「その位置にある要素の読み上げ」に変わり、実際の「タップ（実行）」にはダブルタップが必要になります。スワイプは画面上の要素間を順に移動するナビゲーション操作として機能し、三本指スワイプでスクロールを行います👆

つまり、タッチスクリーンという視覚前提のインターフェースが、VoiceOverやTalkBackによって**まったく異なる操作体系に再定義される**のです。この変容は、OSレベルの支援技術がアプリケーション層に対して行っている最も劇的な介入の一つと言えます🔄

開発者がカスタムジェスチャー（ピンチ、長押し、カスタムスワイプなど）を多用するWebアプリケーションを設計する場合、これらのジェスチャーがVoiceOverやTalkBackの操作と競合しないかを検証する必要があります。WCAG 2.2の達成基準2.5.1（ポインタのジェスチャー）は、マルチポイントやパスベースのジェスチャーに対してシングルポインタの代替手段を求めていますが、その背景にある理由は、こうした支援技術の操作体系との共存を考慮したものです✋

この操作体系の変容を文章で読むだけでは、その劇的さは伝わりきりません。実際に視覚障碍者の方がスマートフォンを流暢に操作する姿を見ることで、「タッチスクリーンは視覚がなければ使えない」という開発者の無意識の思い込みが打ち砕かれるのです🔨

# ARIA属性の「正しい使い方」は当事者が知っている

WAI-ARIA（Web Accessibility Initiative – Accessible Rich Internet Applications）は、HTMLのセマンティクスだけでは表現しきれないインタラクティブなUI要素のアクセシビリティ情報を補完するための仕様です。`role`、`aria-label`、`aria-expanded`、`aria-live` といった属性を通じて、動的なWebアプリケーションの状態をスクリーンリーダーに伝達します🏷️

しかしARIAには有名な原則があります。[WAI-ARIA Authoring Practices](https://www.w3.org/WAI/ARIA/apg/)の冒頭に記されている「ARIAの第一のルール」——**ネイティブHTML要素で実現できるものにARIA属性を使うな**——です。`<button>` 要素の代わりに `<div role="button">` を使うと、キーボード操作やフォーカス管理を自前で実装しなければならず、仕様との差異が生まれるリスクが高まります⚡

問題は、ARIA属性の付与における微妙な誤りが、自動テストではほぼ検出できないという点にあります。たとえば、`aria-live="polite"` を指定した動的コンテンツの更新が、実際のスクリーンリーダーでは期待どおりのタイミングで読み上げられないケースは少なくありません。ブラウザとスクリーンリーダーの組み合わせによる実装差異も存在し、仕様書の記述だけを頼りに実装したコードが実環境で正しく動作する保証はないのです📋

当事者によるデモンストレーションは、このような**仕様と実装のギャップ**を可視化するもっとも直接的な手段です🔍

# 「共感」を超えた「理解」へ ― デモンストレーションの技術的意義

当事者のデモンストレーションの価値を「共感を生む」という文脈だけで語るのは、その意義の半分しか捉えていません。技術者にとっての本質的な価値は、以下の三つの技術的な理解が得られることにあります🎯

## ユーザーエージェントの多様性の実感

Web標準の世界では、ブラウザを「ユーザーエージェント」と呼びます。しかし実際には、**スクリーンリーダーもまたユーザーエージェント**です。ブラウザのレンダリングエンジンがHTMLを視覚表現に変換するのと同様に、スクリーンリーダーはアクセシビリティツリーを聴覚的・触覚的な表現に変換します。デモンストレーションを見ることで、開発者は「ユーザーエージェントの多様性」を実感として理解し、**マルチユーザーエージェント対応**という視点を自然に獲得できます🌐

## 操作効率の基準値の更新

熟練したスクリーンリーダーユーザーの操作速度は、読み上げ速度を大幅に上げた状態で、見出しジャンプやランドマークナビゲーションを駆使し、驚くほど高速にページ内を移動することがあります。この「熟練したユーザーの操作パターン」を知ることは、パフォーマンス最適化の指標と同様に、**アクセシビリティの品質を測る際の現実的な基準値**を開発者に提供します📊

## テストケースの具体化

「スクリーンリーダーでテストする」という抽象的な知識は、デモンストレーションを通じて具体的なテストシナリオに変換されます。どのキー操作でナビゲートするのか、どのような順序で情報を取得するのか、どの場面でつまずくのか——これらの情報は、手動テストの設計において極めて実用的な知見となります🧪

# 今回のセッションで伝えたいこと

筆者が今回のセッションタイトルに「実際」という語を入れたのは、**理論と実装の橋渡し**を目指しているからです🌉

これまで、WCAG 2.2の体系的な解説、CI/CDにおける自動テストの構築、そして自身の骨折体験を通じたアクセシビリティの考察など、さまざまな角度からWebアクセシビリティについて発信してきました。しかし、**当事者の実際の操作を開発者が直接観察する**というアプローチは、上記のいずれとも異なる独自の価値を持っていると感じています🌟

自動テストは「仕様への適合」を検証できますが、「ユーザー体験の質」は検証できません。ガイドラインの学習は「何をすべきか」を教えてくれますが、「なぜそれが必要なのか」の体感的な理解は与えてくれません。当事者の体験談は「困難の存在」を伝えてくれますが、「具体的にどの要素がどう読み上げられるか」というレベルの詳細は含まれないことがほとんどです📖

だからこそ今回のセッションでは、当事者のデモンストレーションを通じて、これらの手法ではカバーしきれない**「実装が実際のユーザー体験としてどう結実するか」**を参加者の皆さんに直接体感していただきたいのです🔑

# おわりに

Webアクセシビリティにおける最大の障壁の一つは、開発者と当事者の間に存在する**経験の非対称性**です。開発者は自分の作ったWebサイトを自分の目で見て、自分の手で操作し、自分の感覚で品質を判断します。しかしその判断は、晴眼者としての認知モデルに基づくものであり、スクリーンリーダーユーザーの体験とは根本的に異なるフレームワークの上に成り立っています🤔

WCAG 2.2の仕様を読むこと、axe-coreで自動テストを回すこと、自分でスクリーンリーダーを起動してみること——これらはすべて重要です。しかし、**日常的にスクリーンリーダーを使って生活している方の操作を直接観察すること**は、それらとは質的に異なる理解をもたらします🎯

今回の.NETラボ勉強会のセッションを通じて、参加する開発者一人ひとりにとって、アクセシビリティの理解を「知識」から「実感」へと変える契機にしたいと考えています🙏
