## はじめに

昨今、PoCではバイブコーディングによって開発コストがずいぶん下がってきています。

「バイブコーディング」とは、AIツール（GitHub CopilotやCursor、Clineなど）を活用して、人間とAIが協調しながらコーディングを進めるスタイルのことです。

**本番開発でもAIによる開発をどんどん取り入れて開発効率をあげていこう！** というのが今回のテーマです。

## さすがに、クラス設計は人間駆動で

AIは強力ですが、**クラス設計は人間がやったほうがいい**です。

### なぜ人間が設計すべきか？

#### 🔧 保守は人間が担当する

後の保守は主に**人間が担当することになる**ので、**人が分かりやすいデザインパターンを選択することが大事**です。

AIが生成した「動くけど理解しづらい」コードよりも、人間が設計した「読みやすく保守しやすい」コードのほうが、長期的なプロジェクト運用では圧倒的に有利です。

#### 🧩 複雑なシステムではコンテキストが大きすぎる

複雑なシステムにおいて、**デザインパターンの選択はコンテキストが大きすぎてAIにはまだ無理がある**のが現状です。

- 既存システムとの整合性
- チームのスキルセット
- 将来の拡張計画
- ビジネス要件の優先度

これらを総合的に判断してパターンを選ぶのは、現時点では人間のほうが得意です。

### 推奨するアプローチ

1. **大枠のクラス設計を人間が実装**する
2. その設計を**Copilot-Instructions**に落とし込む
3. 詳細な実装はAIに任せる

こうすることで、AIがプロジェクトの設計方針を理解した上でコードを生成してくれます。

## お気に入りのツールでバイブコーディングしようぜ

VSCodeでのGitHub Copilotなど、使い慣れたツールで始めましょう。

### Copilot-Instructionsの環境整備が重要

**Copilot-Instructions**やPromptファイルの環境整備はきちんとしておきましょう。以下のような内容を記載すると効果的です：

| 項目 | 説明 |
|------|------|
| プロジェクト特有のルール | 命名規則、ディレクトリ構成など |
| 言語特有のルール | コーディングスタイル、フォーマット |
| LLMにまだ学習されてないと思われる言語の新しい文法 | 最新のC# 13やTypeScript 5.xの機能など |
| 使用するフレームワーク | 特定のライブラリの使い方 |
| さまざまな作業手順 | デプロイ手順、テスト手順など |

### Copilot-Instructions自体もAIで作れる！

Copilot-Instructions自体も、AIで作らせることができます。

#### 作成の流れ

1. **大枠のクラス設計を人間が作成**する
2. その**クラスファイルをAIに読み込ませる**
3. 設計ドキュメントやInstructionsを**GitHub CopilotなどのAIに生成させる**
4. **人間がレビュー**して足りないところを指摘
5. 指摘に基づいてAIに修正させるか、人間が加筆訂正して完成

このように、設計の「核」は人間が作り、その周辺のドキュメント整備はAIに任せることで、効率的に環境を整えられます。

## モデルごとのプロンプトの書き方

AIモデルによって特性が異なるため、プロンプトの書き方も変えると効果的です。

### Anthropic（Claude）の場合

Anthropicのモデルは**指示していなくても自律的にいろんなことをする傾向**が強いです。

- 思ってもないことをすることもある
- **禁止することを多めのプロンプトを書くと成功率が高い**

```markdown
# やってはいけないこと
- ファイルを勝手に削除しないでください
- 確認なしにリファクタリングしないでください
- テストなしに本番コードを変更しないでください
```

### OpenAI（GPT）の場合

OpenAIのモデルは逆に**指示してることしかしない傾向**が強いです。

- **確実にやってほしいことを細かく書いたほうが成功する可能性が強い**
- ただし、あまり分量が多くても時々忘れる可能性はあり
- （最近のモデルはかなり改善されてる）

```markdown
# 必ず実行すること
- 変更前に必ずテストを実行してください
- エラーハンドリングを必ず追加してください
- コメントを日本語で書いてください
```

### モデルの使い分け

#### ✅ 高性能モデルが向いているケース（GPT-5、Claude Opus、Gemini Pro）

- **複雑なアルゴリズムの設計・実装**
- **難解なバグの根本原因分析**
- **アーキテクチャの検討**
- **複数ファイルにまたがるリファクタリング計画**
- **テストケースの網羅的な設計**

#### ❌ 軽量モデルで十分なケース（GPT-5 mini、Claude Haiku、Gemini Flash）

- 単純なコード補完
- 定型的なボイラープレート生成
- ドキュメントやコメントの作成
- 簡単なリファクタリング
- 日常的なチャットでの質問

### 使い分けのコツ

```
💡 GitHub Copilotでの使い分けポイント

日常的なコーディング → GPT-5 mini / Claude Haiku 4.5 / Raptor mini
  → プレミアムリクエストを消費しない or 少ない

バランスの良い作業 → Claude Sonnet 4.5 / Gemini 2.5 Pro
  → 品質とコストのバランスが良い

最高品質が必要な時 → GPT-5.1 / Claude Opus 4.5
  → 複雑な問題に深い思考で対応
```

バイブコーディングにおいても、**日常的なコーディングは軽量モデル**で、**設計や難しい問題は高性能モデル**という使い分けが効果的です。
