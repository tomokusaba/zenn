## はじめに 🧭

AI エージェントという言葉を聞く機会が増えてきました。チャットで質問に答えるだけではなく、目的を理解し、必要な情報を探し、ツールを呼び出し、場合によっては業務システムへアクションします。そんなソフトウェアの形が、かなり現実的な選択肢になっています。

一方で、組織として導入しようとすると「どのツールを使うか」よりも前に考えることがたくさんあります。誰の業務に適用するのか。どのデータにアクセスさせるのか。失敗したときに誰が止めるのか。監査ログは残るのか。そもそも、どの段階から本番利用と呼ぶのか。

私はこのあたりを考えるとき、Azure Cloud Adoption Framework（CAF）の見方がかなり役立つと感じています。CAF はクラウド導入のためのフレームワークですが、AI エージェント導入にも「組織として順番に考えるべきこと」を整理するレンズとして使えます☁️

本記事では、CAF を知っている方に向けて、AI エージェント導入を CAF の各フェーズに対応づけて考えてみます。単なるツール紹介ではなく、**何を先に決めるべきか、どう段階的に広げるか、どこにガバナンスを置くか**に主眼を置きます。

:::message
この記事は「AI エージェントの実装手順」ではなく、「組織として AI エージェントを導入する順番」を考えるための記事です。
:::

## まず AI エージェントをどう捉えるか 🤖

この記事では、AI エージェントを「LLM（大規模言語モデル）を使って目的達成のためにタスクを計画・実行するソフトウェア的な主体」くらいの粒度で捉えます。

Microsoft の AI agent adoption guidance では、AI agent は生成 AI モデルを使って入力を解釈し、問題を推論し、適切なアクションを決める柔軟なソフトウェアプログラムとして説明されています。また、構成要素として generative AI model（生成 AI モデル）、instructions（指示）、retrieval（検索）、actions（アクション）、memory（記憶）が挙げられています。

つまり、単に「社内文書を検索して回答するチャットボット」よりも少し広い概念です。検索して答えるだけなら Retrieval-Augmented Generation（RAG）的なアプリケーションでも十分な場合があります。しかしエージェントは、状況に応じてどの知識を使うか、どのツールを呼ぶか、次に何をするかを決める方向に進みます。

この違いは、便利さの違いであると同時に、リスクの違いでもあります。情報を返すだけのシステムと、業務システムに変更を加えるシステムでは、必要な統制が変わります。ここを曖昧にしたまま「AI エージェントを導入しよう」と言うと、期待値も責任範囲も膨らんでしまいます。

AI agent adoption guidance では agent types として productivity agents、action agents、automation agents が紹介されています。私はこの分類を、導入段階の成熟度としても見ると分かりやすいと感じています。

| 種類 | 主な役割 | 導入時に見るポイント |
|------|----------|----------------------|
| 📚 Productivity agents | 情報の検索・要約・意思決定支援 | どの情報源を信頼するか、回答の根拠をどう示すか |
| 🛠️ Action agents | 定義された業務アクションの実行 | 権限、承認、取り消し、監査ログをどう設計するか |
| 🔁 Automation agents | 複数ステップの自律的な処理 | 停止条件、例外処理、人間へのエスカレーションをどう置くか |

最初から automation agents を狙うと、技術より先に組織設計で詰まりやすいです。まずは productivity agents で価値とリスクを学び、action agents で権限と監査を固め、automation agents は十分に統制できる領域から始めます。この順番が現実的ではないでしょうか。

AI エージェントは単なる自動化ツールではなく、権限を持って業務に関与し得る主体です。だからこそ、導入・統制・運用をまとめて扱える CAF の観点で整理すると、議論の抜け漏れを見つけやすくなります。

## CAF の 7 つの methodology と AI 導入の関係 ☁️

CAF overview では、Cloud Adoption Framework は Strategy、Plan、Ready、Adopt、Govern、Secure、Manage の 7 つの methodology で構成されると説明されています。Strategy、Plan、Ready、Adopt は foundational methodologies（基礎的な方法論）、Govern、Secure、Manage は operational methodologies（運用上の方法論）です。

なお、CAF の AI adoption シナリオでは、AI Strategy、AI Plan、AI Ready、Govern AI、Secure AI、Manage AI の 6 ステップとして整理されています。本記事では、CAF 全体の 7 つの methodology を AI エージェント導入のレンズとして使い、Adopt も含めて対応づけて考えます。

AI エージェント導入でも、この流れはかなり自然に当てはまります。ただし、AI エージェントでは「モデルが何を返すか」だけでなく「エージェントが何をしてよいか」まで考える必要があります。そのため、Govern / Secure / Manage は後工程ではなく、最初から並走させるべきものとして扱うのがよさそうです。

CAF の良いところは、技術導入を「作る」だけで終わらせないところです。AI エージェントも同じで、PoC から組織の中で使われ続ける仕組みにするには、戦略、計画、環境、導入、統制、保護、運用が必要になります。

先に全体像を表にすると、次のような対応になります。

| CAF methodology | AI エージェント導入で考えること |
|-----------------|----------------------------------|
| 🎯 Strategy | どの成果を改善するのか、任せてよい判断と任せない判断は何か |
| 📝 Plan | ユースケース、データ、スキル、PoC、Responsible AI をどう計画するか |
| 🏗️ Ready | エージェントを安全に動かす環境、境界、接続、landing zone をどう整えるか |
| 🚀 Adopt | 読み取り専用から承認付きアクションへ、権限をどう段階的に広げるか |
| 🛡️ Govern | ユースケース登録、責任者、例外、監査、停止条件をどう決めるか |
| 🔒 Secure | 最小権限、監査可能性、circuit breaker、独立したガードレールをどう置くか |
| ⚙️ Manage | 利用状況、品質、コスト、データ鮮度、振る舞いをどう継続運用するか |

## Strategy: 何を達成したいかを決める 🎯

Strategy で最初に決めたいのは、「AI エージェントで何を自動化するか」ではなく「どのビジネス成果を改善したいか」です。

AI エージェントは魅力的な技術なので、つい「問い合わせ対応をエージェント化したい」「社内申請を自動化したい」のように、手段から話が始まりがちです。しかし CAF の Strategy の考え方に寄せるなら、まずは business drivers（ビジネス上の推進要因）と cloud outcomes（クラウド導入で得たい成果）を対応づけるように、AI エージェントで狙う成果を言語化する必要があります。

たとえば、次のような問いから始めると整理しやすいです。

- 🧑‍💼 **従業員体験**: 情報検索や定型作業にかかる時間を減らしたいのか
- 🧾 **業務品質**: 手入力や転記によるミスを減らしたいのか
- 🧪 **開発生産性**: 調査、実装、レビュー、運用対応のサイクルを短くしたいのか
- 🧭 **意思決定**: 散らばった情報を集約して判断材料を早く出したいのか

ここで重要なのは、AI エージェントに「任せたい作業」と「任せてはいけない判断」を分けることです。問い合わせの一次回答は任せても、契約条件の最終判断は人間が行います。障害対応の調査は任せても、本番環境への変更は承認を必須にします。こうした境界は、実装より先に決めるべきです。

AI エージェントは、業務の中で小さな主体のように振る舞います。だからこそ、導入の最初に「この主体は何のために存在するのか」を明確にしておく必要があります。ここで定義した成果が、以降のユースケース選定、権限設計、運用設計の判断基準になります。

## Plan: スキル、データ、PoC、Responsible AI を計画に入れる 📝

Plan では、Strategy で定義した成果を、ユースケース、データ、リスク、スキルの観点で実行可能なロードマップに落とし込みます。Microsoft の Plan for AI adoption では、AI スキル評価、AI maturity、データ資産の棚卸し、PoC、Responsible AI を早期に組み込む観点が示されています。

AI エージェント導入で特に大事なのは、ユースケースを「価値が高い順」だけで並べないことです。価値が高くても、データが整っていない、権限設計が難しい、失敗時の影響が大きい場合は、最初の導入対象としては重すぎるかもしれません。

私なら、出力が要約や候補提示にとどまるか、外部送信や本番更新を伴うか、人間が確認して実行できるか、という軸で優先順位を付けます。社内ナレッジ検索、会議メモの整理、サポート担当者向けの回答案作成などは、retrieval や instructions の品質を学びやすい一方で、外部への直接影響を抑えやすい領域です。

また、Plan の時点で Responsible AI を後回しにしないことも重要です。あとから「監査ログを付けよう」「禁止事項を追加しよう」とすると、エージェントの設計そのものを作り直すことになりがちです。

:::message alert
AI エージェントの PoC は「動いたら成功」ではなく、「どの条件なら安全に使えるかを学べたら成功」と考える方がよいです。
:::

Plan では、技術ロードマップだけでなく、スキル習得、データ整備、責任分界、評価方法まで含めて計画します。ここが曖昧だと、PoC は楽しくても本番導入で止まります。

次に、その計画を安全に試せる環境と境界を Ready で整えます。

## Ready: エージェントを置く場所と境界を整える 🏗️

Ready では、AI ワークロードを載せる環境を整えます。AI Ready では、AI governance（AI ガバナンス）、networking（ネットワーク）、reliability（信頼性）、Azure landing zone、internal / internet-facing workloads（内部向け / インターネット向けワークロード）の分離などが扱われています。

Plan は「何を、どの順番で試すか」を決める段階です。一方で Ready は、それを安全に動かす場所と境界を整える段階です。

AI エージェント導入では、この「境界」の考え方がとても重要です。エージェントは知識にアクセスし、ツールを呼び出し、場合によっては外部ユーザーと対話します。社内規程を参照する従業員向けエージェントと、顧客からの問い合わせを受ける外部向けエージェントでは、使えるデータ、認証、ネットワーク、失敗時の影響が変わります。

Azure landing zone を使っている組織であれば、AI ワークロードも通常のアプリケーション landing zone に載せるという考え方は自然です。ただし、AI エージェントはモデル、検索、データ、API、監視が組み合わさるため、リソース単体ではなく「エージェントが触れる範囲」を単位に境界を設計する必要があります。

どのサブスクリプション、どの管理グループ、どのネットワーク境界、どのポリシー継承の中でエージェントを動かすのか。Ready でここを決めておくと、後続の Adopt がかなり進めやすくなります。

## Adopt: 小さく始めて、権限を段階的に広げる 🚀

Adopt では、実際にワークロードを導入します。AI エージェントの場合、私は「対象業務を広げる」よりも先に「権限を段階的に広げる」発想が大事だと思っています。

最初は、エージェントを読み取り専用にします。ユーザーの質問に答える、関連文書を探す、作業手順を提示する。次に、下書きや提案を作らせます。チケットの説明文、メールの案、作業計画などです。そして、十分に評価できた範囲で、承認付きのアクションに進みます。

| 段階 | エージェントに許可すること | 人間の関与 |
|------|----------------------------|------------|
| 1 | 📖 読み取り、検索、要約 | 回答を利用するか人間が判断 |
| 2 | 📝 下書き、候補作成、分類 | 人間が確認して送信・登録 |
| 3 | ✅ 承認付きアクション | 人間の承認後に実行 |
| 4 | 🔁 条件付き自動実行 | 例外時に人間へエスカレーション |

この段階設計は、AI agent adoption guidance の productivity agents、action agents、automation agents の考え方とも相性がよいです。いきなり自律実行を目指すのではなく、まずは生産性支援から始め、業務アクションを限定し、最後に自動化範囲を広げていきます。

Adopt の中で忘れたくないのは、評価観点です。想定質問、例外ケース、権限不足のケース、不適切な入力、古いデータが混ざったケースなどを用意し、どの条件で失敗するかを観察する必要があります。答えられないときは曖昧に回答しない。権限がないときは実行しない。確信が持てないときは人間へ渡す。こうした振る舞いを設計に反映していきます。

ここでの Adopt は「段階的に導入する原則」を扱います。後半の「私ならこう段階導入する」では、この原則を CAF の観点に戻してロードマップとして整理します。

## Govern: ガバナンスは境界と例外の設計 🛡️

Govern は、AI エージェント導入で最も早く話し合うべき領域の一つです。エージェントは人間の業務に入り込み、データとアクションをつなぐ存在だからです。

ここでいうガバナンスは、「全部を中央で承認する」という意味ではありません。現場が安全に試せるように、境界と例外処理を明確にすることだと思います。

- 🧭 **ユースケース登録**: どの部門が、何の目的で、どのデータを使うのか
- 🧑‍⚖️ **責任者**: エージェントの出力、アクション、改善判断の owner は誰か
- 🗂️ **データ分類**: 利用してよいデータ、追加審査が必要なデータは何か
- 🔐 **権限モデル**: エージェント自身の権限と利用者の権限をどう組み合わせるか
- 🧾 **監査**: 誰が、いつ、何を入力し、何を参照・実行したか
- 🚦 **停止条件**: どの異常を検知したら止めるか、誰が再開を判断するか

AI エージェントは便利だからこそ、野良エージェント化しやすいです。CAF 的に考えるなら、ガバナンスはブレーキではなくガードレールです。現場が進めるために走ってよい車線を作り、逸脱したときに検知できるようにします。この考え方が大切です。

ただし、ルールを決めるだけでは守り切れません。次に、技術的に強制・検知・停止する Secure の観点を見ていきます。

## Secure: Agentic AI safeguards を前提にする 🔒

Secure では、AI エージェント特有のリスクを考えます。Responsible AI in Azure workloads では、agentic AI safeguards として auditability（監査可能性）、RBAC、circuit breaker（異常時に処理を止める仕組み）、robust data ingress / egress（堅牢なデータ入出力）、data validation / integrity（データ検証と整合性）、independent guardrails（独立したガードレール）などの観点が示されています。

AI エージェントのセキュリティは、単にモデル API を守るだけでは足りません。入力、検索対象、ツール呼び出し、出力、外部送信、実行結果のすべてが保護対象になります。

私が Secure で最初に置きたいのは、次の 3 つです。

1. **最小権限**: エージェントに人間以上の権限を持たせない
2. **監査可能性**: エージェントの判断材料とアクションを追跡できるようにする
3. **停止可能性**: 危険な兆候があるときに circuit breaker 的に止められるようにする

:::message
AI エージェントは「便利な自動化ツール」である前に、「権限を持って動くソフトウェア」です。権限、監査、停止の 3 点を最初から設計に入れることが重要です。
:::

また、ガードレールはエージェントの instructions だけに閉じない方がよいです。アクセス制御、ネットワーク境界、データ検証、外部の監視、承認フローなど、独立した層で守る設計が必要です。

Govern、Secure、Manage は似た言葉に見えるかもしれません。私は、Govern は組織ルールと例外設計、Secure は技術的な強制・検知・停止、Manage は導入後の継続運用を扱うものとして分けて考えています。

## Manage: 作って終わりではなく、振る舞いを運用する ⚙️

Manage では、AI エージェントを継続的に運用します。従来のアプリケーション運用では、可用性、性能、コスト、エラー率などを見ます。AI エージェントでは、それに加えて「振る舞い」を見続ける必要があります。

たとえば、どの部門で使われているか、期待した根拠に基づいて回答できているか、実行した処理が業務上正しい結果になったかを見ます。あわせて、コストが想定内か、誤回答や誤操作が起きていないか、grounding data（根拠データ）が古くなっていないかも確認します。

AI エージェントの品質は、業務ルール、社内文書、API、利用者の期待、モデルやサービス側の更新によって変わります。そのため、一度リリースしたら安定ではなく、継続的に評価し、改善し、必要なら権限を戻す運用が必要です。

## 私ならこう段階導入する 🪜

ここまでを踏まえると、私なら「Strategy を明確化する → 読み取り専用 PoC で学ぶ → 下書き・提案に広げる → 承認付きアクションを試す → 限定的な自動化に進む」という順番で進めます。

| 段階 | 主に関係する CAF 観点 | ねらい |
|------|----------------------|--------|
| 1. Strategy 明確化 | Strategy / Govern | 成果、責任者、任せない判断を決める |
| 2. 読み取り専用 PoC | Plan / Ready / Govern | データ、検索品質、利用者の期待値を学ぶ |
| 3. 下書き・提案 | Adopt / Manage | 人間が確認できる範囲で業務に組み込む |
| 4. 承認付きアクション | Adopt / Secure / Manage | 権限、監査、承認、取り消しを検証する |
| 5. 限定的な自動化 | Govern / Secure / Manage | 停止条件と運用体制がある範囲で広げる |

ポイントは、技術的な高度さよりも組織の学習速度を優先することです。人間が確認できる範囲から始め、観測できる状態を作り、止められる仕組みを持ったうえで、少しずつ任せる範囲を広げる。これが現実的な進め方だと思います。

## おわりに 🌱

AI エージェントは、これから多くの業務に入り込んでいくと思います。だからこそ、導入の議論をツール比較だけで終わらせるのはもったいないです。

CAF の観点で見ると、AI エージェント導入には Strategy、Plan、Ready、Adopt だけでなく、Govern、Secure、Manage が最初から深く関わります。特に、エージェントが actions を持ち始めると、ガバナンスとセキュリティは後付けできません。

私自身、AI エージェントを考えるときは「どこまで賢くできるか」よりも、「どこまで安全に任せられるか」を先に考えたいです。任せる範囲を小さく始め、観測し、学び、必要なら戻します。その積み重ねが、組織としての AI エージェント導入を現実的なものにしていくのだと思います。

CAF はクラウド導入のフレームワークですが、AI エージェントのような新しい技術を組織に根付かせるときにも、とても良い地図になります。CAF を使うことで、AI エージェントを単なる機能導入ではなく、成果、環境、統制、運用を含む組織変革として扱いやすくなります。地図を持ったうえで、一歩ずつ進めていきたいですね🧭
