# 日本語推敲スキル

このスキルは、日本語で書かれた技術文章（特に Zenn / ブログ記事）を読みやすく整えるためのチェックリストと修正フォーマットを提供します。文章のスタイルそのものは [`write-zenn-article`](../write-zenn-article/SKILL.md) §3 に従い、その上で**機械的に検出可能な問題**を本スキルで潰します。

---

## 1. 推敲チェックリスト

以下の観点を **上から順に** 全文に対して適用してください。

### 1.1 文体の統一

- ✅ **です・ます調 / である調を混在させない**。Zenn 著者は基本「です・ます調」（[`write-zenn-article` §3.1](../write-zenn-article/SKILL.md)）。
- ✅ 一人称は「私」で統一。「筆者」「僕」「俺」を混ぜない。
- ✅ 体言止めは **idea 系記事や強調したい箇所のみ**。技術手順では避ける。

### 1.2 表記ゆれ

代表的な揺れ（左を**推奨**側に統一）。プロジェクト内の既存記事を参照して優勢な側に寄せる。

| 推奨 | 避ける | 備考 |
|------|--------|------|
| ユーザー | ユーザ | JIS Z 8301 / Microsoft 日本語スタイルに準拠 |
| サーバー | サーバ | 同上 |
| コンピューター | コンピュータ | 同上 |
| ブラウザー | ブラウザ | 同上 |
| Web | web / WEB | 固有名詞・略語は表記を固定 |
| GitHub | Github / github | 固有名詞は公式表記 |
| .NET | dotnet / Dotnet | 文中での名称は `.NET`、識別子は `dotnet` |
| 例えば | たとえば | 文書内で統一されていればどちらでも可 |
| および | 及び | ひらく |
| ～できます | ～する事ができます | 冗長表現を避ける |

### 1.3 冗長表現の削減

| 修正前 | 修正後 |
|--------|--------|
| ～することができます | ～できます |
| ～という事ができます | ～と言えます |
| ～を行います | ～します |
| ～の場合においては | ～の場合は |
| ～において | ～で |
| ～に関して | ～について |
| まず最初に | まず / 最初に |
| 後で後悔する | 後悔する |

### 1.4 一文の長さ

- 目安: **60〜80 文字 / 1 文**。100 文字を超えたら分割を検討。
- 「が、」で文が続いている場合は要注意（逆接でないなら句点で切る）。

### 1.5 受動態の濫用

- 「〜される」「〜られる」が 2 文連続 / 1 段落に 3 つ以上あれば能動態に書き換えを検討。
- 例: 「設定が読み込まれます」→「設定を読み込みます」

### 1.6 句読点・記号

- 句読点は **「、」と「。」**（カンマ・ピリオドを混在させない）。
- 1 文中の読点は **2〜3 個まで**。多い場合は文を分割。
- 半角英数の前後には **半角スペース 1 つ**を入れる（例: `Azure を使う`）。
- 括弧は **全角（）** を基本に、コード・識別子内は半角 `()` を使う。
- 三点リーダは「…」（U+2026）を使う。「...」（半角ドット 3 つ）は避ける。
- 感嘆符・疑問符の後ろは全角スペース 1 つ（行末は不要）。

### 1.7 カタカナ語と英字の使い分け

- **広く定着しているカタカナ語**: ユーザー、サーバー、デプロイ、ログ、コンテナー
- **製品名・固有名詞は原綴**: Azure、Kubernetes、PostgreSQL、Visual Studio Code
- **略語は初出時に展開**: 「CI（継続的インテグレーション）」のように一度だけ展開

### 1.8 専門用語の初出時の補足

- 初めて出る専門用語には**和訳または英語表記を併記**する（[`write-zenn-article` §3.1](../write-zenn-article/SKILL.md)）。
- 例: 「アクチュエーションポイント（入力が確定する深さ）」

### 1.9 見出し・箇条書き

- 見出しは原則 **体言止め または 「〜する」「〜とは」** 形式。文末に句点は付けない。
- 箇条書きは **粒度を揃える**。1 項目だけ長文になっていないか確認。
- 番号付きリストは「手順」のみで使う。並列概念は `-` を使う。

### 1.10 「ら抜き」「い抜き」「さ入れ」

- ❌ 見れる / 食べれる / 出れる → ✅ 見られる / 食べられる / 出られる
- ❌ ～してます → ✅ ～しています
- ❌ ～させていただきます（過剰敬語） → ✅ ～します / ～いたします

## 2. 出力フォーマット

推敲結果は **「修正前 / 修正後 / 理由」3 列の Markdown 表** で返します。理由欄では本スキル §1 の項番（例: §1.3 冗長表現）を参照してください。

```markdown
## 推敲結果

### サマリ
- チェック項目: 文体 / 表記ゆれ / 冗長 / 受動態 / 句読点 / その他
- 検出件数: N 件（うち必須修正 N 件、推奨修正 N 件）

### 修正提案

| # | 修正前 | 修正後 | 理由 |
|---|--------|--------|------|
| 1 | この機能は内部で処理が行われます。 | この機能は内部で処理を行います。 | §1.5 受動態の濫用 |
| 2 | サーバを起動する事ができます。 | サーバーを起動できます。 | §1.2 表記ゆれ + §1.3 冗長 |
| 3 | まず最初にAzureにログインします。 | まず Azure にログインします。 | §1.3 冗長 + §1.6 半角英数前後の空白 |

### diff 形式（任意）

\`\`\`diff
- この機能は内部で処理が行われます。
+ この機能は内部で処理を行います。
\`\`\`
```

長い文章を扱う場合は、セクション（`##`）ごとに上記表を分けて出力します。

---

## 3. やってはいけないこと

- 元文の意味を変える書き換え（事実関係・主張・ニュアンスを保つ）
- 著者の文体（一人称「私」、対話的な呼びかけ、絵文字運用）を消す
- 専門用語を勝手に平易な言葉に置換する（読者層は技術者前提）
- コードブロック・引用ブロック内の文章を書き換える（コメントを除く）
- フロントマター（YAML）の内容を書き換える
